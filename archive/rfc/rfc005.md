## RFC-005: Blueprint Workflow, Versioning, Library, Favorites, and Moderation

### Status

Proposed

### Summary

Implement Blueprint creation, validation, editing, immutable versioning, approval request, moderation, library browsing, details, and favorite handling.

### Problem Statement

BEE uses Blueprints as reusable experiment specifications. To preserve reproducibility, public or submitted Blueprints must not be silently mutated. The system must support governance through approval and moderation.

### Goals

- Implement Blueprint wizard.
- Implement Blueprint validation.
- Implement Blueprint library and detail pages.
- Implement favorites.
- Implement immutable versioning.
- Implement approval and moderation workflow.
- Make approved Blueprints selectable for experiments.

### Non-Goals

- No experiment execution yet.
- No Public Hub listing implementation yet, except exposing approved Blueprints later.

### Implementation Plan

#### Stage 1: Implement blueprint wizard steps and validator

Implement BlueprintWizardView and BlueprintWizardController for Blueprint creation. Add wizard steps for basic metadata, atomic indicator selection, reference architecture selection, parameter range definition, and review. Add BlueprintValidator rules for required name, supported indicators, valid parameter ranges, supported reference architecture, and safe architecture settings. Persist valid Blueprints as DRAFT records owned by the authenticated User.

Refs: WizardView, BlueprintWizardView, BlueprintWizardController, BlueprintValidator, Blueprint

#### Stage 2: Implement blueprint library detail and favorites

Implement BlueprintsLibraryView, BlueprintDetailView, BlueprintsLibraryController, BlueprintController, and FavoriteBlueprint persistence. Add owned and favorited tabs, Blueprint detail sections for metadata, indicator configuration, architecture configuration, approval state, version number, parent Blueprint reference, and owner information. Add favorite and unfavorite actions for accessible Blueprints. Use AccessControlService to prevent unauthorized viewing of private DRAFT Blueprints.

Refs: BlueprintsLibraryView, BlueprintDetailView, BlueprintsLibraryController, BlueprintController, FavoriteBlueprint, AccessControlService

#### Stage 3: Implement immutable blueprint versioning service

Implement VersioningService for Blueprint lifecycle integrity. Allow never-submitted DRAFT Blueprints to be edited in place. When a previously submitted, approved, rejected, or disapproved Blueprint is edited, create a new DRAFT Blueprint version with parent_id pointing to the original, incremented version number, copied pipeline specification, reset approval state, and preserved owner visibility. Integrate VersioningService with BlueprintWizardController and BlueprintApprovalController so public or previously reviewed artifacts are never silently mutated.

Refs: VersioningService, BlueprintWizardController, BlueprintApprovalController, Blueprint

#### Stage 4: Implement approval moderation state flow

Implement Blueprint approval and moderation. Add request approval action from BlueprintDetailView, PENDING transition through BlueprintApprovalController, and staff review through BlueprintModerationView. Add approve, reject, and disapprove actions with role checks from AccessControlService. Approved Blueprints become selectable in ExperimentWizardController and visible through PublicHubController. Rejected Blueprints remain owner-visible and can be revised through VersioningService. Disapproved Blueprints are removed from public visibility and create a remediation path.

Refs: BlueprintDetailView, BlueprintModerationView, BlueprintApprovalController, AccessControlService, VersioningService, PublicHubController

#### Stage 5: Add wizard validation and draft persistence tests

Add tests for BlueprintWizardView, BlueprintWizardController, BlueprintValidator, and Blueprint persistence. Verify valid Blueprint creation, missing name rejection, unsupported indicator rejection, empty parameter range rejection, unsupported architecture rejection, successful DRAFT persistence, wizard step navigation, review summary rendering, and backend validation error display.

Tests: Blueprint wizard, validator, DRAFT persistence

#### Stage 6: Add versioning moderation library and favorite tests

Add tests for Blueprint library, detail view, favorites, versioning, and moderation. Verify owned Blueprint listing, favorited Blueprint listing, Blueprint detail access rules, favorite and unfavorite persistence, DRAFT in-place editing, submitted Blueprint edit creating a new DRAFT version, parent_id preservation, version number increment, approval request transition to PENDING, Moderator approval transition to APPROVED, rejection transition to REJECTED, disapproval removal from Public Hub visibility, and approved Blueprint availability for ExperimentWizardController selection.

Tests: VersioningService, BlueprintApprovalController, BlueprintsLibraryController, FavoriteBlueprint

### Pseudocode

```python
def save_blueprint_edit(blueprint_id, updates, user_id):
    existing = blueprint_repo.get_by_id(blueprint_id)

    if existing.user_id != user_id:
        raise ForbiddenError()

    if existing.status == "DRAFT" and existing.submitted_at is None:
        return blueprint_repo.update(existing.id, updates)

    return versioning_service.create_versioned_copy(
        blueprint_id=existing.id,
        config=updates,
    )
```

```python
def moderate_blueprint(blueprint_id, staff, action):
    access_control.require_moderator_or_admin(staff)

    if action == "APPROVE":
        blueprint.status = "APPROVED"
    elif action == "REJECT":
        blueprint.status = "REJECTED"
    elif action == "DISAPPROVE":
        blueprint.status = "DISAPPROVED"

    blueprint_repo.save(blueprint)
```
