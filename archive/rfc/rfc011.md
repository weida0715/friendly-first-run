## RFC-011: Public Hub, Documentation, Logs, and Exports

### Status

Proposed

### Summary

Implement public discovery and export features. Authenticated users should browse enabled users, successful experiments, eligible models, and approved Blueprints. Users should also be able to download completed experiment logs and metrics.

### Problem Statement

BEE’s usability contribution includes public discovery and reuse. The Public Hub must expose only safe and approved artifacts while keeping private or failed records hidden.

### Goals

- Implement Public Hub tabs.
- Implement public user profiles.
- Implement public artifact navigation.
- Implement documentation pages.
- Implement completed experiment CSV exports.

### Non-Goals

- No unauthenticated public browsing unless explicitly enabled later.
- No external sharing links.

### Design Classes Covered

- `PublicHubView`
- `PublicHubController`
- `UserProfileView`
- `ExperimentDetailView`
- `ModelDetailView`
- `BlueprintDetailView`
- `DocumentationView`
- `DocumentationController`
- `LogsDownloadController`

### Implementation Plan

#### Stage 1: Implement public discovery tabs and api

Implement PublicHubController and PublicHubView with tabs for Users, Experiments, Models, and Blueprints. Retrieve enabled Users, successful public Experiments, eligible Models, and approved Blueprints through repositories. Add search and filters by owner, artifact type, metric, Blueprint status, and date. Use AccessControlService to ensure private records, disabled users, draft Blueprints, pending Blueprints, rejected Blueprints, disapproved Blueprints, failed experiments, and unauthorized models are not exposed.

Refs: PublicHubController, PublicHubView, AccessControlService, User, Experiment, Model, Blueprint

#### Stage 2: Add public profile and artifact navigation

Add navigation from PublicHubView to UserProfileView, ExperimentDetailView, ModelDetailView, and BlueprintDetailView. Implement public profile summaries showing enabled user metadata and public artifacts. Ensure detail navigation respects visibility rules so a public list item cannot be used to access private Experiment, Model, or Blueprint records.

Refs: PublicHubView, UserProfileView, ExperimentDetailView, ModelDetailView, BlueprintDetailView

#### Stage 3: Implement documentation controller and views

Implement DocumentationController and DocumentationView. Add documentation listing and documentation detail rendering for user guide, experiment workflow, Blueprint workflow, model ranking guide, Public Hub guide, administrator guide, and system limitations. Keep DocumentationView accessible to authenticated users and structure content so it can be linked from the dashboard and module pages.

Issue: RFC-011; Refs: DocumentationController, DocumentationView, AccessControlService

#### Stage 4: Implement experiment metrics and logs download

Implement LogsDownloadController and export endpoints for completed Experiments. Add CSV download support for Model metric columns and ExperimentLog records, model parameter summaries, and split boundary metadata. Add download actions in ExperimentDetailView. Use AccessControlService to ensure only authorized users can export private experiment artifacts.

Refs: LogsDownloadController, ExperimentDetailView, Model, ExperimentLog

#### Stage 5: Add visibility search and navigation tests

Add tests for PublicHubController and PublicHubView. Verify enabled users are visible, disabled users are hidden, successful public experiments are visible, failed or private experiments are hidden, eligible models are visible, private models are hidden, approved Blueprints are visible, draft pending rejected and disapproved Blueprints are hidden, search and filters return correct subsets, and navigation to UserProfileView, ExperimentDetailView, ModelDetailView, and BlueprintDetailView respects access rules.

Tests: Public Hub visibility, search, filters, navigation

#### Stage 6: Add documentation and experiment export tests

Add tests for DocumentationController, DocumentationView, LogsDownloadController, and export endpoints. Verify documentation listing and detail rendering, CSV export authorization, completed experiment export success, incomplete experiment export rejection, confusion metric CSV structure, evaluation metric CSV structure, experiment log CSV structure, and unauthorized export rejection.

Tests: documentation, CSV export, logs, metrics

### Pseudocode

```python
def get_public_hub_data(user, tab, filters):
    if tab == "users":
        return user_repo.list_enabled(filters)

    if tab == "experiments":
        return experiment_repo.list_successful_public(filters)

    if tab == "models":
        return model_repo.list_public_eligible(filters)

    if tab == "blueprints":
        return blueprint_repo.list_approved(filters)
```

```python
def download_experiment_logs(experiment_id, user, export_type):
    experiment = experiment_repo.get_by_id(experiment_id)

    if not access_control.can_view_experiment(user, experiment):
        raise ForbiddenError()

    if experiment.status != "COMPLETED":
        raise ConflictError("Only completed experiments can be exported")

    rows = export_service.build_rows(experiment_id, export_type)
    return csv_response(rows)
```

### Acceptance Tests

- Public Hub shows enabled users only.
- Public Hub shows approved Blueprints only.
- Public Hub hides failed experiments.
- Public Hub hides private models.
- CSV export works for completed experiments.
