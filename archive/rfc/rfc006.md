## RFC-006: BTCUSDT Spot 1m Kline Retrieval, Storage, Update, and Charting

### Status

Proposed

### Summary

Implement BTCUSDT spot 1m kline retrieval using Binance Connector. Store retrieved klines in the database, update the cache on every experiment run, and visualize price data using TradingView Lightweight Charts.

### Problem Statement

BEE experiments require BTCUSDT spot market data. The data must be refreshed before each experiment run, persisted locally for reproducibility, and displayed in the frontend for user context.

### Goals

- Retrieve BTCUSDT spot 1m klines.
- Normalize Binance response data.
- Store klines in `BTCUSDTKline`.
- Upsert candles without duplicates.
- Refresh data before every experiment run.
- Load experiment data from local cache.
- Render BTCUSDT chart in the frontend.

### Non-Goals

- No live trading.
- No multi-symbol support.
- No order placement.
- No on-chain metrics.

### Design Classes Covered

- `BTCUSDTKline`
- `MarketDataService`
- `MarketDataRepository`
- `DefaultExperimentExecutor`
- `ExperimentExecutor`
- `DashboardView`
- `ExperimentWizardView`
- `ExperimentDetailView`

### Implementation Plan

#### Stage 1: Add binance btcusdt 1m kline client

Implement Binance Connector infrastructure for BTCUSDT spot 1m kline retrieval. Add BinanceKlineClient with symbol fixed to BTCUSDT and default interval fixed to 1m for the primary experiment dataset. Add request parameter validation for start time, end time, pagination limit, retry behavior, and response normalization. Convert Binance kline arrays into BTCUSDTKline-compatible candle objects containing timestamp, open, high, low, close, volume, close_time, quote_volume, trade_count, taker_buy_base_volume, and taker_buy_quote_volume where available.

Refs: BTCUSDTKline, MarketDataService, DefaultExperimentExecutor, infrastructure/binance

#### Stage 2: Implement btcusdt kline upsert cache service

Implement MarketDataService and MarketDataRepository behavior for storing and updating BTCUSDT 1m klines. Add upsert logic so repeated retrievals update existing candles and insert missing candles without duplicate timestamp records. Add cache query methods for retrieving candles by date range and interval. Add refresh metadata such as created_at and updated_at. Ensure BTCUSDTKline storage remains the local source used by the experiment pipeline after each refresh.

Refs: MarketDataService, MarketDataRepository, BTCUSDTKline

#### Stage 3: Refresh btcusdt 1m data before every experiment run

Integrate BTCUSDT 1m market data refresh into experiment execution preparation. Before DefaultExperimentExecutor loads data for an Experiment, call MarketDataService.refresh_btcusdt_1m for the Experiment date range. After refresh succeeds, load the dataset from MarketDataRepository instead of directly from Binance. This ensures every run updates the BTCUSDT spot 1m cache while keeping the split-first pipeline reproducible from persisted database candles.

Refs: DefaultExperimentExecutor, ExperimentExecutor, MarketDataService, MarketDataRepository, BTCUSDTKline, Experiment

#### Stage 4: Add tradingview btcusdt price chart component

Add BTCUSDT price visualization using TradingView Lightweight Charts. Create a reusable chart component under frontend/components/charts and integrate it into DashboardView, ExperimentWizardView, and ExperimentDetailView. Add backend chart data endpoint backed by MarketDataRepository. Convert BTCUSDTKline records into chart-compatible time and price series. Add loading, empty, and error states so chart rendering remains stable when data is unavailable or refresh fails.

Refs: DashboardView, ExperimentWizardView, ExperimentDetailView, BTCUSDTKline

#### Stage 5: Add binance connector and cache upsert tests

Add tests for BTCUSDT 1m market data retrieval and persistence. Mock Binance Connector responses and verify BTCUSDT symbol enforcement, 1m interval enforcement, date range validation, response normalization, BTCUSDTKline object creation, MarketDataRepository upsert behavior, duplicate timestamp prevention, updated_at changes on repeated candles, correct retrieval by date range, and safe service error handling when Binance retrieval fails.

Tests: BinanceKlineClient, MarketDataService, MarketDataRepository, BTCUSDTKline

#### Stage 6: Add execution refresh and chart rendering tests

Add tests confirming BTCUSDT 1m data refresh occurs before experiment execution. Verify DefaultExperimentExecutor calls MarketDataService before loading candles, refreshed candles are read from MarketDataRepository, repeated experiment runs update existing candles instead of inserting duplicates, and execution can proceed using cached data after refresh. Add frontend tests for the TradingView chart in DashboardView, ExperimentWizardView, and ExperimentDetailView, covering loading, empty, successful render, and error states.

Tests: DefaultExperimentExecutor, MarketDataService, BTCUSDT chart component

### Pseudocode

```python
def refresh_btcusdt_1m(start, end):
    candles = binance_client.fetch_klines(
        symbol="BTCUSDT",
        interval="1m",
        start=start,
        end=end,
    )

    market_data_repo.upsert_klines(candles)
    return RefreshSummary(inserted=n_inserted, updated=n_updated)
```

```python
class DefaultExperimentExecutor(ExperimentExecutor):
    def load_data(self, config):
        market_data_service.refresh_btcusdt_1m(
            start=config.start_date,
            end=config.end_date,
        )

        candles = market_data_repo.get_klines(
            symbol="BTCUSDT",
            interval="1m",
            start=config.start_date,
            end=config.end_date,
        )

        if len(candles) < MIN_REQUIRED_CANDLES:
            raise ExecutionError("Insufficient BTCUSDT 1m candles")

        return candles_to_dataframe(candles)
```
