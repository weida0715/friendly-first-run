## RFC-012: Security Hardening, Error Handling, and Deployment Configuration

### Status

Proposed

### Summary

Add CSRF protection, secure session settings, ownership guards, structured errors, deployment scripts, environment configuration, and deployment smoke tests.

### Problem Statement

Before final acceptance testing, the application must have secure state-changing requests, consistent error responses, safe private-resource access, and reliable local production startup.

### Goals

- Add CSRF protection.
- Add secure cookie settings.
- Add ownership guards.
- Add structured API errors.
- Add deployment scripts.
- Add environment examples.
- Add smoke tests.

### Non-Goals

- No cloud deployment automation unless needed later.
- No HTTPS certificate automation unless needed later.

### Design Classes Covered

- `AccessControlService`
- `SessionService`
- `AuthenticationController`
- `UserController`
- `BlueprintWizardController`
- `BlueprintApprovalController`
- `ExperimentWizardController`
- `ModelController`
- `JobController`
- `LogsDownloadController`

### Implementation Plan

#### Stage 1: Add csrf protection and secure session settings

Add Flask-WTF CSRF protection for state-changing backend endpoints. Configure secure session cookie settings including HttpOnly, SameSite, and Secure flags where environment permits. Add frontend CSRF token retrieval and inclusion for POST, PUT, PATCH, and DELETE requests. Apply CSRF protection to AuthenticationController logout, UserController updates, BlueprintWizardController submissions, BlueprintApprovalController moderation actions, ExperimentWizardController submissions, ModelController favorite actions, JobController cancellation, and LogsDownloadController export requests where applicable.

Refs: Flask-WTF, SessionService, protected controllers

#### Stage 2: Add ownership guards and structured errors

Strengthen AccessControlService and controller-level guards. Add reusable ownership checks for User, Blueprint, Experiment, Model, FavoriteModel, FavoriteBlueprint, Job metadata, and export artifacts. Add structured API errors for validation failures, authentication failures, authorization failures, missing resources, conflict errors, external Binance failures, database failures, and worker failures. Update frontend views to display safe user-facing errors through alerts and toasts without exposing stack traces or internal configuration.

Refs: AccessControlService, protected controllers, Boundary Layer error states

#### Stage 3: Add local production configuration and run scripts

Add deployment-ready configuration for local production testing. Add backend run scripts, frontend build scripts, environment examples, database migration commands, worker startup command, Redis connection settings, PostgreSQL connection settings, Binance Connector configuration, CORS settings, and port documentation. Document default ports for Next.js frontend, Flask backend, PostgreSQL database, Redis broker, and worker process. Ensure configuration supports the final website with prototype-matching frontend, BTCUSDT 1m kline refresh, experiment execution, model ranking, Public Hub, and user management.

Refs: deployment configuration, Next.js, Flask, PostgreSQL, Redis, worker, Binance Connector

#### Stage 4: Add csrf rbac injection and structured error tests

Add security tests for CSRF protection, RBAC, ownership guards, SQL injection resistance, and structured errors. Verify missing CSRF tokens are rejected, valid CSRF tokens are accepted, unauthenticated users cannot access protected APIs, Normal Users cannot call staff endpoints, Moderators cannot call Administrator-only endpoints, users cannot access another user's private Experiment or Model, SQL injection-like inputs do not alter queries, invalid payloads return structured errors, Binance failures return safe errors, and frontend views show safe user-facing messages.

Tests: CSRF, RBAC, ownership, injection resistance, structured errors

#### Stage 5: Add build migration worker and smoke tests

Add deployment smoke tests. Verify frontend production build completes, backend starts with production-like settings, database migrations apply cleanly, Redis connection is available, worker process can start, health-check endpoint responds, frontend can call backend, BTCUSDT kline refresh command can run with mocked Binance data, and a queued experiment can be processed by the worker in a controlled test environment.

Tests: production build, migrations, worker startup, smoke flow

### Pseudocode

```ts
async function apiPost(path, body) {
  const csrfToken = await getCsrfToken()

  return fetch(API_BASE_URL + path, {
    method: "POST",
    credentials: "include",
    headers: {
      "Content-Type": "application/json",
      "X-CSRFToken": csrfToken,
    },
    body: JSON.stringify(body),
  })
}
```

### Acceptance Tests

- State-changing request without CSRF is rejected.
- Valid CSRF token is accepted.
- Unauthorized access is rejected.
- SQL injection-like input does not alter query behavior.
- Frontend production build works.
- Worker starts successfully.
- Health-check works in production-like configuration.
