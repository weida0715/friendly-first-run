## RFC-002: Strict ERD-Aligned Domain Entities, Database Schema, ORM, and Repositories

### Status

Proposed

### Summary

Implement the persistence foundation for BEE **strictly according to the Chapter 4 3NF ERD diagram**. The database implementation must use the exact entity/table names, primary key names, foreign key names, column names, enum values, nullable rules, and constraints shown in the ERD. Domain and repository code may use Pythonic naming internally, but the physical database schema and ORM mappings must preserve the ERD naming style.

### Problem Statement

The application needs a stable persistence layer before authentication, Blueprints, experiments, workers, and model rankings can function. Because Chapter 4 already defines the normalized 3NF ERD, RFC-002 must not introduce extra persistence tables, renamed snake_case tables, additional metric tables, job tables, server session tables, or columns that are not in the ERD. If later RFCs need queue metadata, server sessions, or extended evaluation metrics, those must be implemented outside this strict ERD schema unless the ERD is formally revised.

### Strict ERD Rule

The database schema for RFC-002 must follow this Chapter 4 ERD exactly:

- Entity/table names must be **singular PascalCase**.
- Primary keys must use the ERD names, such as `UserID`, `BlueprintID`, `ExperimentID`, `ModelID`, and `ExperimentLogID`.
- Foreign keys must use the referenced entity naming style, such as `UserID`, `BlueprintID`, `ExperimentID`, and `ModelID`.
- Columns must use **PascalCase** with no underscores.
- The BTCUSDT market cache must be the fixed `BTCUSDTKline` table with `Timestamp` as the primary key.
- The database implementation must not create snake_case duplicates such as `users`, `blueprints`, `experiments`, `models`, or `btcusdt_klines`.
- The database implementation must not create additional RFC-002 tables such as `JobRecord`, `ServerSession`, `ExperimentEvaluationMetrics`, or `ExperimentConfusionMetrics`, because these are not present in the supplied ERD.
- Extended metrics required later must be stored through the ERD-approved fields, especially `Model` metric columns and `ExperimentLog.Metrics` JSON.

### Goals

- Implement only the ERD-approved domain entities.
- Implement immutable value objects used across services without creating extra database tables.
- Create SQLAlchemy ORM mappings that preserve the exact ERD table and column names.
- Create migrations matching the ERD exactly.
- Implement repository classes around the ERD schema.
- Add transaction handling through Unit of Work.
- Add repository, ORM, constraint, and naming-convention tests.

### Non-Goals

- No user login behavior yet.
- No Blueprint workflow behavior yet.
- No experiment execution behavior yet.
- No Binance API retrieval yet.
- No additional database tables outside the ERD.
- No queue/session persistence tables unless the ERD is formally revised.
- No separate `Model metric columns` or `ExperimentLog.Metrics JSON` table.

### Strict Database Schema

#### User

```text
User
  UserID       INT PK Auto Increment
  Username     VARCHAR(12) UNIQUE NOT NULL
  Email        VARCHAR(255) UNIQUE NOT NULL
  PasswordHash VARCHAR(60) NOT NULL
  Name         VARCHAR(100) NOT NULL
  Role         ENUM('User','Moderator','Admin') NOT NULL
  Status       ENUM('Enabled','Disabled','Pending') NOT NULL
  CreatedAt    DATETIME NOT NULL
  UpdatedAt    DATETIME NOT NULL
```

#### Blueprint

```text
Blueprint
  BlueprintID    INT PK Auto Increment
  UserID         INT FK -> User.UserID
  Name           VARCHAR(100) NOT NULL
  Description    TEXT NULLABLE
  Indicators     JSON
  Features       JSON
  Architecture   JSON
  ApprovalState  ENUM('Draft','Pending','Approved','Rejected') NOT NULL
  SubmittedAt    DATETIME NULLABLE
  Version        INT NOT NULL
  ParentID       INT FK -> Blueprint.BlueprintID NULLABLE
  CreatedAt      DATETIME NOT NULL
  UpdatedAt      DATETIME NOT NULL

Constraints:
  UNIQUE(UserID, Name, Version)
  CHECK(ParentID IS NULL OR ParentID <> BlueprintID)
```

#### Experiment

```text
Experiment
  ExperimentID        INT PK Auto Increment
  UserID              INT FK -> User.UserID
  BlueprintID         INT FK -> Blueprint.BlueprintID
  Name                VARCHAR(100) NOT NULL
  Description         TEXT NULLABLE
  Interval            ENUM('1m','5m','15m','1h','4h','1d') NOT NULL
  StartDate           DATE NOT NULL
  EndDate             DATE NOT NULL
  TrainSplit          NUMERIC(3,2) CHECK NOT NULL
  ValSplit            NUMERIC(3,2) CHECK NOT NULL
  TestSplit           NUMERIC(3,2) CHECK NOT NULL
  ParameterOverrides  JSON NULLABLE
  Status              ENUM('Queued','Running','Completed','Failed','Cancelled') NOT NULL
  Progress            NUMERIC(5,2) NULLABLE
  CurrentStage        VARCHAR(50) NULLABLE
  EtaSeconds          INT NULLABLE
  Success             BOOLEAN NULLABLE
  CreatedAt           DATETIME NOT NULL
  CompletedAt         DATETIME NULLABLE

Constraints:
  CHECK(TrainSplit + ValSplit + TestSplit = 1.00)
  CHECK(ValSplit >= 0.10 AND TestSplit >= 0.10)
```

### Repository Scope

Repositories must use the strict ERD names in their ORM mappings while exposing convenient domain methods to the service layer.

```text
UserRepository
BlueprintRepository
ExperimentRepository
ModelRepository
ExperimentLogRepository
FavoriteModelRepository
FavoriteBlueprintRepository
MarketDataRepository
UnitOfWork
```

No `MetricsRepository`, `JobRepository`, or `SessionRepository` should be introduced in RFC-002 unless the ERD is revised. Model-level metrics must use the `Model` table fields. Per-row execution or evaluation metrics must use `ExperimentLog.Metrics` JSON.

### Pseudocode

#### ValidationResult value object

```python
@dataclass(frozen=True)
class ValidationResult:
    ok: bool
    errors: dict[str, list[str]]

    @staticmethod
    def success():
        return ValidationResult(ok=True, errors={})

    @staticmethod
    def failure(field, message):
        return ValidationResult(ok=False, errors={field: [message]})
```

#### UnitOfWork using only ERD repositories

```python
class UnitOfWork:
    def __enter__(self):
        self.session = SessionLocal()
        self.users = UserRepository(self.session)
        self.blueprints = BlueprintRepository(self.session)
        self.experiments = ExperimentRepository(self.session)
        self.models = ModelRepository(self.session)
        self.experiment_logs = ExperimentLogRepository(self.session)
        self.favorite_models = FavoriteModelRepository(self.session)
        self.favorite_blueprints = FavoriteBlueprintRepository(self.session)
        self.market_data = MarketDataRepository(self.session)
        return self

    def __exit__(self, exc_type, exc, tb):
        if exc:
            self.session.rollback()
        else:
            self.session.commit()
        self.session.close()
```

#### BTCUSDTKline upsert using Timestamp primary key

```python
def upsert_btcusdt_klines(candles):
    for candle in candles:
        existing = session.get(BTCUSDTKlineORM, candle.Timestamp)

        if existing is None:
            session.add(BTCUSDTKlineORM(**to_erd_columns(candle)))
        else:
            existing.Open = candle.Open
            existing.High = candle.High
            existing.Low = candle.Low
            existing.Close = candle.Close
            existing.Volume = candle.Volume
```

### Acceptance Tests

- Domain objects can be instantiated without Flask dependencies.
- Migrations create exactly the ERD-approved tables and no additional RFC-002 tables.
- Table names are singular PascalCase exactly as defined in the ERD.
- Column names are PascalCase exactly as defined in the ERD.
- Enum values match the ERD exactly.
- `User.Username` and `User.Email` are unique.
- `Blueprint` enforces `UNIQUE(UserID, Name, Version)`.
- `Blueprint.ParentID` references `Blueprint.BlueprintID` and cannot equal its own `BlueprintID`.
- `Experiment` enforces split sum and minimum validation/test split constraints.
- `BTCUSDTKline.Timestamp` is the primary key and prevents duplicate candles.
- `FavoriteModel` and `FavoriteBlueprint` enforce composite primary keys.
- Repository CRUD operations work through the strict ERD schema.
- Failed UnitOfWork transaction rolls back.
- Relationships between `User`, `Blueprint`, `Experiment`, `Model`, and `ExperimentLog` are valid.
