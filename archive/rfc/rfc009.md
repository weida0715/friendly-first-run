## RFC-009: Split-First Experiment Execution and Parameter Permutations

### Status

Proposed

### Summary

Implement the core experiment execution engine. The pipeline must refresh BTCUSDT 1m data, load persisted candles, split chronologically first, compute indicators per split, compose features per split, generate parameter permutations, train/evaluate models, and persist metrics.

### Problem Statement

The academic contribution of BEE depends on reproducible, split-first execution. If indicators or features are computed before splitting, look-ahead bias can occur. The execution pipeline must structurally prevent this.

### Goals

- Implement `ExperimentExecutor` Template Method.
- Implement `DefaultExperimentExecutor`.
- Refresh BTCUSDT 1m data before execution.
- Load from local cache.
- Split train/validation/test chronologically.
- Compute indicators per split.
- Generate exhaustive parameter permutations.
- Store one `Model` per permutation.
- Store evaluation metrics and logs.
- Support reproducible repeated runs.

### Non-Goals

- No live trading.
- No brokerage integration.
- No multi-symbol experiments.

### Implementation Plan

#### Stage 1: Add experiment executor template method

Implement ExperimentExecutor as the abstract Template Method base class for BEE experiment execution. Add a final execute(experiment_id) workflow that calls load_data, create_temporal_splits, compute_indicators_per_split, compose_features_per_split, transform_targets, scale_features, train_model, run_internal_evaluation, and log_artifacts in a fixed order. Add DefaultExperimentExecutor as the concrete executor for the first implementation. Use ExecutionResult, SplitResult, TrainedModel, and EvaluationResult value objects to pass structured data between stages.

Refs: ExperimentExecutor, DefaultExperimentExecutor, ExecutionResult, SplitResult, TrainedModel, EvaluationResult

#### Stage 2: Implement btcusdt refresh load and chronological split stages

Implement the first DefaultExperimentExecutor stages. Refresh BTCUSDT 1m data through MarketDataService, load BTCUSDTKline records from MarketDataRepository, validate sufficient candles, convert candles to DataFrame, and create chronological train validation test splits using Experiment split ratios. Store split boundary metadata for reproducibility and later export. Ensure no indicator, feature, target, scaling, model, or evaluation stage runs before chronological splitting completes.

Refs: DefaultExperimentExecutor, MarketDataService, MarketDataRepository, BTCUSDTKline, Experiment

#### Stage 3: Implement per split indicators features targets and scaling

Implement indicator computation, feature composition, target transformation, and feature scaling inside DefaultExperimentExecutor. Add initial support for SMA, RSI, MACD, and ATR where defined by Blueprint indicators. Compute indicators separately for train, validation, and test splits. Compose features only within each split. Transform binary directional targets per split. Fit scalers using training data only and apply trained scaler parameters to validation and test splits. Prevent validation and test data from influencing training indicator, feature, target, or scaler state.

Refs: DefaultExperimentExecutor, Blueprint, Experiment, SplitResult

#### Stage 4: Implement blueprint executor factory and logreg executor

Implement BlueprintExecutorFactory and ReferenceArchitectureExecutor strategy support. Add LogRegBinaryExecutor as the first concrete architecture executor for binary directional modeling. Resolve Blueprint architecture configuration and Experiment parameter overrides into executable model settings. Keep architecture-specific training behavior inside ReferenceArchitectureExecutor implementations so DefaultExperimentExecutor can train models through a stable strategy interface.

Refs: BlueprintExecutorFactory, ReferenceArchitectureExecutor, LogRegBinaryExecutor, DefaultExperimentExecutor, Blueprint, Experiment

#### Stage 5: implement parameter permutation and model artifact persistence

Implement exhaustive parameter permutation generation from Blueprint parameter ranges and Experiment parameter_overrides. Generate one concrete configuration per Cartesian product combination. For each permutation, train and evaluate one Model artifact, persist Model parameters and metrics, link the Model to the parent Experiment and Blueprint, and prevent duplicate Model records for identical parameter hashes within the same Experiment. Update Experiment progress after each permutation completes.

Refs: BlueprintExecutorFactory, DefaultExperimentExecutor, Experiment, Blueprint, Model

#### Stage 6: Persist evaluation metrics logs and reproducibility artifacts

Persist execution outputs for reproducibility. Store Model metric columns and ExperimentLog records, split boundary metadata, parameter hash, run seed, BTCUSDT kline refresh timestamp, data range, model parameter summary, and artifact metadata for each completed Experiment. Add LogsDownloadController support for later CSV export of confusion metrics, evaluation metrics, experiment logs, and split metadata. Update Experiment status, success flag, completed_at timestamp, and progress at the end of execution.

Refs: Model, ExperimentLog, LogsDownloadController, DefaultExperimentExecutor, Experiment, Model

#### Stage 7: Add template order and temporal integrity tests

Add tests for ExperimentExecutor and DefaultExperimentExecutor temporal integrity. Verify execute() calls stages in the required order, BTCUSDT 1m data refresh occurs before loading from cache, train validation test splits are chronological, no future candles appear in earlier splits, indicators are computed after splitting, features are composed per split, target transformation does not leak future data, scaler fitting uses training data only, and execution fails safely when insufficient candles exist.

Tests: ExperimentExecutor, DefaultExperimentExecutor, MarketDataService, BTCUSDTKline, SplitResult

#### Stage 8: Add permutation metrics and reproducibility tests

Add tests for permutation generation, model artifact creation, evaluation metrics, and reproducibility. Verify BlueprintExecutorFactory resolves architecture configuration, LogRegBinaryExecutor trains through ReferenceArchitectureExecutor, parameter overrides narrow or fix Blueprint ranges without mutating Blueprint, expected Model count equals the Cartesian product of parameter values, one Model is stored per permutation, duplicate parameter hashes are prevented, Model metric columns and ExperimentLog.Metrics JSON are stored, ExperimentLog records are generated, and repeated runs with identical data and configuration produce consistent outputs.

Tests: BlueprintExecutorFactory, ReferenceArchitectureExecutor, LogRegBinaryExecutor, Model, metrics, logs

### Pseudocode

```python
class ExperimentExecutor:
    def execute(self, experiment_id):
        config = self.load_config(experiment_id)

        data = self.load_data(config)
        splits = self.create_temporal_splits(data, config)
        splits = self.compute_indicators_per_split(splits, config)
        splits = self.compose_features_per_split(splits, config)
        splits = self.transform_targets(splits, config)
        splits = self.scale_features(splits, config)

        model = self.train_model(splits, config)
        results = self.run_internal_evaluation(model, splits.test)
        self.log_artifacts(config, model, results)

        return results
```

```python
def generate_permutations(blueprint_params, overrides):
    effective_params = merge_ranges(blueprint_params, overrides)

    keys = effective_params.keys()
    values = [effective_params[key] for key in keys]

    for combination in product(*values):
        yield dict(zip(keys, combination))
```

### Acceptance Tests

- Execution sequence cannot skip split-first ordering.
- BTCUSDT refresh runs before data loading.
- No future data appears in training split.
- Indicators are computed after splitting.
- Expected model count equals parameter Cartesian product.
- Metrics are stored.
- Repeated same configuration produces consistent output.
