## RFC-003: Authentication, Sessions, RBAC, and User Management

### Status

Proposed

### Summary

Implement secure account registration, login, logout, server-managed sessions, route protection, role-based access control, and staff user management.

### Problem Statement

BEE is a multi-user research platform. The system must protect private experiments and Blueprints, allow staff moderation, and prevent unauthorized access to administrative functions.

### Goals

- Implement registration.
- Implement password hashing.
- Implement login and logout.
- Implement server-managed sessions.
- Implement current-user context.
- Implement protected frontend routes.
- Implement Normal User, Moderator, and Administrator access rules.
- Implement user management screens and APIs.

### Non-Goals

- No Blueprint moderation yet.
- No experiment creation yet.
- No Public Hub yet.

### Implementation Plan

#### Stage 1: Implement registration form and create user api

Implement user registration from RegistrationView to AuthenticationController. Add the frontend registration form using Shadcn/ui inputs for name, username, email, password, and password confirmation. Add backend registration endpoint that normalizes username, validates username length and lowercase alphanumeric format, checks duplicate username and email using UserRepository, hashes the password before persistence, creates a User entity with Normal User role and Enabled status, and returns the created user summary.

Refs: RegistrationView, AuthenticationController, UserRepository, User

#### Stage 2: Implement login logout and session lifecycle

Implement login, logout, and server-managed sessions. Add LoginView form submission to AuthenticationController. Add credential verification using email and password, password hash comparison, disabled account rejection, SessionService session creation, secure session cookie response, current-user endpoint, and logout session destruction. Integrate AccessControlService for session verification and user identity extraction.

Refs: LoginView, LandingPageView, DashboardView, AuthenticationController, SessionService, AccessControlService, User

#### Stage 3: Add protected route handling and current user provider

Add frontend protected route handling using the current user endpoint. Implement an AuthProvider that loads the authenticated user through AccessControlService-backed API responses. Protect DashboardView, ExperimentListView, ExperimentWizardView, ExperimentDetailView, ModelsRankingsView, ModelsLibraryView, ModelDetailView, BlueprintsLibraryView, BlueprintDetailView, BlueprintWizardView, PublicHubView, DocumentationView, UserProfileView, UserManagementView, SystemManagementView, BlueprintModerationView, and JobDetailView.

Refs: AccessControlService and protected Boundary Layer views

#### Stage 4: Implement rbac and user management workflows

Implement role-based access control and staff user management. Add AccessControlService methods for checking authentication, staff access, moderator access, administrator access, ownership, and role hierarchy. Implement UserController methods for listing users, viewing user profile, creating Normal User accounts, enabling and disabling accounts, updating roles, resetting passwords, and deleting users. Implement UserManagementView with search, role/status badges, create-user dialog, enable/disable actions, reset password action, Administrator-only role update, and Administrator-only delete action.

Refs: UserController, UserManagementView, UserProfileView, SystemManagementView, AccessControlService, User

#### Stage 5: Add registration login logout and session tests

Add tests for RegistrationView, LoginView, AuthenticationController, SessionService, AccessControlService, and User. Verify valid registration, duplicate username rejection, duplicate email rejection, invalid username rejection, password mismatch rejection, password hashing, successful login, invalid login, disabled account rejection, current-user retrieval, logout session destruction, unauthenticated protected route redirection, and authenticated protected route rendering.

Tests: registration, login, logout, sessions, protected routes

#### Stage 6: Add rbac and staff management tests

Add tests for UserController, UserManagementView, AccessControlService, and User. Verify Normal Users cannot access staff pages, Moderators can create Normal Users and enable or disable Normal User accounts, Moderators cannot delete users or update Administrator roles, Administrators can list all users, create users, update roles, reset passwords, enable accounts, disable accounts, and delete users. Verify role changes immediately affect AccessControlService decisions and disabled users cannot authenticate.

Tests: RBAC, user management, account state transitions

### Pseudocode

```python
def authenticate(email, password):
    user = user_repo.get_by_email(email)

    if user is None:
        raise AuthError("Invalid credentials")

    if user.status != "ENABLED":
        raise AuthError("Account disabled")

    if not verify_password(password, user.password_hash):
        raise AuthError("Invalid credentials")

    session_id = session_service.create_server_session(
        user_id=user.id,
        role=user.role,
    )

    return LoginResponse(session_id=session_id, user=safe_user(user))
```

```python
def can_manage_user(actor, target_user):
    if actor.role == "ADMIN":
        return True

    if actor.role == "MODERATOR":
        return target_user.role == "NORMAL_USER"

    return False
```

### Acceptance Tests

- Valid user can register.
- Password is not stored in plaintext.
- Valid user can log in.
- Invalid password is rejected.
- Disabled user cannot log in.
- Normal User cannot access staff pages.
- Moderator cannot perform Administrator-only actions.
- Administrator can manage users.
