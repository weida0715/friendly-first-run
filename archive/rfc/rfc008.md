## RFC-008: Queue, Worker, Jobs, Cancellation, and System Management

### Status

Proposed

### Summary

Implement asynchronous experiment execution through a queue and background worker. Users should be able to submit experiments without blocking the web request, view job status, and cancel eligible jobs.

### Problem Statement

Experiments may run for a long time because BTCUSDT data refresh, split-first processing, parameter permutation generation, training, and evaluation can be expensive. These jobs must run asynchronously.

### Goals

- Implement queue abstraction.
- Implement Redis/RQ adapter or equivalent worker runner.
- Enqueue experiments after creation.
- Track job status.
- Implement job detail view.
- Implement cancellation strategy.
- Implement system queue view for administrators.

### Non-Goals

- No actual model training in this issue.
- No model ranking in this issue.

### Implementation Plan

#### Stage 1: Implement job queue abstraction and redis adapter

Implement the queue abstraction for experiment jobs. Add JobQueue protocol, RedisJobQueue adapter, QueueService orchestration methods, JobSpecification value object, and QueuePosition value object. QueueService exposes enqueue_job, get_queue_position, remove_job_from_queue, get_active_jobs, and cancel_running_job while hiding Redis/RQ details inside the infrastructure layer.

Refs: QueueService, JobQueue, RedisJobQueue, JobSpecification, QueuePosition

#### Stage 2: Enqueue experiment after successful configuration

Connect ExperimentWizardController to QueueService. After a valid Experiment is created, enqueue an experiment execution job, set Experiment status to QUEUED, persist job metadata through JobRepository, and return queue position to ExperimentWizardView. Add queue status display in DashboardView and ExperimentDetailView so users can see that the experiment has been accepted for background execution.

Refs: ExperimentWizardController, QueueService, JobRepository, ExperimentWizardView, DashboardView, ExperimentDetailView, Experiment

#### Stage 3: Implement experiment worker runner and status updates

Implement the background worker runner and experiment job handler. Add worker bootstrap code, job payload validation, Experiment status transition from QUEUED to RUNNING, safe exception handling, completion transition to COMPLETED, failure transition to FAILED, progress update hooks, and worker logging. Prepare the worker to call DefaultExperimentExecutor for actual execution. Ensure worker code uses repositories and services rather than frontend or controller logic.

Refs: workers, QueueService, ExperimentRepository, DefaultExperimentExecutor, Experiment

#### Stage 4: Implement job detail cancellation and system queue view

Implement JobDetailView, JobController, cancellation flow, and SystemManagementView queue visibility. Add job detail retrieval, owner verification, queue position display, progress display, cancellation action, and administrator active job table. Add JobCancellationHandlerRegistry, CancellableJobStrategy, and ExperimentCancellationHandler so cancellation follows the Strategy pattern from the final design. Use AccessControlService to prevent users from cancelling jobs they do not own.

Refs: JobDetailView, JobController, SystemManagementView, JobCancellationHandlerRegistry, CancellableJobStrategy, ExperimentCancellationHandler, AccessControlService, QueueService

#### Stage 5: Add queue service worker lifecycle tests

Add tests for QueueService, RedisJobQueue, worker runner, and Experiment status transitions. Verify job enqueue, queue position retrieval, queued status persistence, worker job pickup, RUNNING transition, COMPLETED transition, FAILED transition with error details, and repository updates. Mock Redis/RQ where appropriate so tests remain deterministic.

Tests: QueueService, RedisJobQueue, worker lifecycle, Experiment status

#### Stage 6: Add job detail cancellation and system queue tests

Add tests for JobController, JobDetailView, SystemManagementView, JobCancellationHandlerRegistry, CancellableJobStrategy, and ExperimentCancellationHandler. Verify users can view their own job status, cannot view unauthorized jobs, can cancel their own queued jobs, cannot cancel another user's job, Administrator can view active queue, queue limit display renders correctly, and cancellation updates Experiment status and QueueService state.

Tests: JobController, JobDetailView, SystemManagementView, cancellation strategies

### Pseudocode

```python
def submit_experiment(config, user):
    experiment = create_experiment(config, user)

    job_spec = JobSpecification(
        job_type="EXPERIMENT_EXECUTION",
        payload={"experiment_id": experiment.id},
        priority="normal",
    )

    queue_position = queue_service.enqueue_job(job_spec)

    experiment_repo.update_status(
        experiment.id,
        status="QUEUED",
        job_id=queue_position.job_id,
    )

    return {
        "experiment_id": experiment.id,
        "job_id": queue_position.job_id,
        "queue_position": queue_position.position,
    }
```

```python
def handle_experiment_job(payload):
    experiment_id = payload["experiment_id"]

    experiment_repo.update_status(experiment_id, "RUNNING", progress=0.0)

    try:
        result = default_executor.execute(experiment_id)
        experiment_repo.mark_completed(experiment_id, result.summary)
    except Exception as exc:
        experiment_repo.mark_failed(experiment_id, error=str(exc))
        raise
```

### Acceptance Tests

- Experiment submission creates queued job.
- Worker processes queued job.
- Job status transitions from QUEUED to RUNNING to COMPLETED.
- Failed job becomes FAILED.
- User can cancel own queued job.
- User cannot cancel another user’s job.
- Admin can view active queue.
