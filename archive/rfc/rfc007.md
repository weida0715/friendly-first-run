## RFC-007: Experiment Configuration and Experiment Management

### Status

Proposed

### Summary

Implement the experiment wizard and management pages. Users should create experiments using BTCUSDT 1m data, choose date range, configure train/validation/test splits, select a Blueprint, optionally override parameters, review, submit, and view experiment details.

### Problem Statement

The experiment configuration flow is the main user entry point into BEE’s research workflow. It must enforce split rules, Blueprint access, and experiment-specific overrides while preserving Blueprint immutability.

### Goals

- Implement experiment wizard.
- Implement experiment validation.
- Persist experiment records.
- Store parameter overrides only on the experiment.
- Implement experiment list and detail pages.
- Protect private experiments.

### Non-Goals

- No queue processing in this issue.
- No execution pipeline in this issue.
- No model ranking in this issue.

- `ExperimentWizardView`

### Implementation Plan

#### Stage 1: Implement experiment wizard prototype flow

Implement ExperimentWizardView using the prototype-aligned multi-step flow. Add steps for experiment basics, BTCUSDT data range, train validation test split, Blueprint selection, parameter override configuration, review, and submit. Use WizardView for step navigation and Shadcn/ui components for inputs, cards, tabs, alerts, and review summary. Keep ExperimentWizardView presentation-only and send the final configuration to ExperimentWizardController.

Refs: WizardView, ExperimentWizardView, ExperimentWizardController, Experiment, Blueprint

#### Stage 2: Add experiment validator and split rules

Implement ExperimentValidator for experiment configuration. Validate required name, fixed BTCUSDT dataset scope, supported interval values, date range ordering, train validation test split total, minimum validation split, minimum test split, accessible Blueprint selection, and parameter override constraints. Return ValidationResult objects so ExperimentWizardController can send structured field errors to ExperimentWizardView.

Refs: ExperimentValidator, ValidationResult, ExperimentWizardController, ExperimentWizardView

#### Stage 3: Persist experiment configuration and overrides

Implement ExperimentWizardController creation flow. On valid configuration, create an Experiment linked to the authenticated User and selected Blueprint. Persist name, description, fixed BTCUSDT symbol, 1m interval, start_date, end_date, train_split, val_split, test_split, parameter_overrides, status, progress, and metadata. Ensure parameter_overrides are stored only on the Experiment entity and never mutate the selected Blueprint. Add ownership and access checks through AccessControlService and BlueprintRepository.

Refs: ExperimentWizardController, AccessControlService, ExperimentRepository, BlueprintRepository, User, Blueprint, Experiment

#### Stage 4: Implement experiment list detail and status pages

Implement ExperimentListView and ExperimentDetailView with backend support from ExperimentController. Add user-specific experiment listing, status filtering, experiment summary cards, configuration display, selected Blueprint summary, parameter override summary, progress display, created/completed timestamps, model result links, job detail links, and logs download placeholders. Use AccessControlService to ensure users can only view their own private experiments or public-eligible experiments.

Refs: ExperimentListView, ExperimentDetailView, ExperimentController, AccessControlService, Experiment, Blueprint, Model

#### Stage 5: Add wizard validation and override tests

Add tests for ExperimentWizardView, ExperimentWizardController, and ExperimentValidator. Verify required name validation, fixed BTCUSDT scope, 1m interval enforcement, valid and invalid date ranges, split total validation, minimum validation and test split rules, accessible Blueprint selection, inaccessible Blueprint rejection, valid parameter override acceptance, invalid override rejection, and structured ValidationResult responses. Verify frontend step navigation and validation error rendering.

Tests: ExperimentWizardView, ExperimentWizardController, ExperimentValidator

#### Stage 6: Add experiment persistence list and detail tests

Add tests for Experiment persistence and management pages. Verify valid Experiment creation, correct linkage to User and Blueprint, correct storage of BTCUSDT 1m configuration, parameter_overrides stored only on Experiment, ExperimentListView displays only accessible experiments, status filtering works, ExperimentDetailView displays configuration details correctly, and unauthorized users cannot view another user's private Experiment.

Tests: ExperimentRepository, ExperimentController, ExperimentListView, ExperimentDetailView

### Pseudocode

```python
def validate_experiment(config, user):
    if not config.name:
        return fail("name", "Experiment name is required")

    if config.symbol != "BTCUSDT":
        return fail("symbol", "Only BTCUSDT is supported")

    if config.interval != "1m":
        return fail("interval", "Experiment execution uses BTCUSDT 1m data")

    if config.start_date >= config.end_date:
        return fail("date_range", "Start date must be before end date")

    if config.val_split < 0.10:
        return fail("val_split", "Validation split must be at least 10%")

    if config.test_split < 0.10:
        return fail("test_split", "Test split must be at least 10%")

    if round(config.train_split + config.val_split + config.test_split, 4) != 1.0:
        return fail("splits", "Splits must total 100%")

    if not blueprint_repo.user_can_access(user.id, config.blueprint_id):
        return fail("blueprint_id", "Blueprint is not accessible")

    return success()
```

### Acceptance Tests

- Valid experiment can be created.
- Invalid split ratios are rejected.
- Validation/test split below 10% is rejected.
- Inaccessible Blueprint is rejected.
- Parameter overrides do not mutate Blueprint.
- Experiment list and detail pages display correct information.
