## RFC-004: Prototype-Matching Frontend Implementation

### Status

Proposed

### Summary

Implement the visible frontend so it matches the prototype. This issue focuses on layout, navigation, routes, reusable Shadcn/ui components, dashboard cards, data tables, empty states, loading states, and responsive behavior.

### Problem Statement

The FYP2 deliverable must not only work technically but also look like the planned prototype. The UI should be consistent across all modules and should keep business logic out of presentation components.

### Goals

- Implement the main app shell.
- Implement navigation.
- Implement reusable form, table, dialog, badge, and status components.
- Implement dashboard and module landing pages.
- Add prototype parity tests.

### Non-Goals

- No actual Blueprint submission logic in this issue.
- No experiment execution logic in this issue.
- No model ranking data logic in this issue.

### Design Classes Covered

All Boundary Layer classes, especially:

- `BaseView`
- `WizardView`
- `DashboardView`
- `ExperimentListView`
- `ExperimentWizardView`
- `BlueprintWizardView`
- `ModelsRankingsView`
- `PublicHubView`
- `UserManagementView`

### Implementation Plan

#### Stage 1: Implement prototype aligned app shell navigation and theme

Implement the main frontend application shell to match the prototype layout. Add sidebar or top navigation, page header area, breadcrumb area, content containers, responsive grid layout, typography scale, spacing rules, and theme support where applicable. Apply the layout consistently to LandingPageView, DashboardView, ExperimentListView, BlueprintsLibraryView, ModelsRankingsView, PublicHubView, UserManagementView, SystemManagementView, BlueprintModerationView, and JobDetailView.

Refs: BaseView, LandingPageView, DashboardView, ExperimentListView, PublicHubView

#### Stage 2: Add reusable form table dialog and status components

Add reusable Shadcn/ui-based components for form fields, validation errors, select menus, date inputs, number inputs, cards, tables, tabs, badges, dialogs, dropdown menus, alerts, loading skeletons, empty states, and toast notifications. Use these components across RegistrationView, LoginView, ExperimentWizardView, BlueprintWizardView, ModelsRankingsView, ModelsLibraryView, BlueprintsLibraryView, PublicHubView, UserManagementView, BlueprintModerationView, and JobDetailView.

Refs: Shadcn/ui components and Boundary Layer views

#### Stage 3: Implement dashboard and module landing pages

Implement DashboardView and landing pages for experiments, Blueprints, models, Public Hub, documentation, users, system management, and jobs. Add dashboard cards for recent experiments, queued jobs, top models, approved Blueprints, BTCUSDT market status, and quick actions. Add placeholder integration points for DashboardController, ExperimentController, ModelsRankingsController, BlueprintsLibraryController, PublicHubController, DocumentationController, UserController, and QueueService.

Refs: DashboardView, DashboardController, all module landing views

#### Stage 4: Add prototype parity route and component rendering tests

Add frontend tests for prototype parity and UI reliability. Verify all major routes render, navigation items route correctly, protected pages require authentication, Shadcn/ui components render expected labels, status badges display correct variants, loading states appear during API requests, empty states appear when datasets are empty, dialogs open and close correctly, forms show validation messages, and layouts remain usable on desktop and tablet widths.

Tests: route rendering, component rendering, prototype parity

### Pseudocode

```tsx
function BaseView({ title, description, children, loading, error }) {
  if (loading) return <LoadingSkeleton />
  if (error) return <ErrorAlert message={error.message} />

  return (
    <PageShell>
      <PageHeader title={title} description={description} />
      {children}
    </PageShell>
  )
}
```

```tsx
function StatusBadge({ status }) {
  return <Badge variant={statusVariantMap[status]}>{status}</Badge>
}
```

### Acceptance Tests

- All routes render.
- Navigation works.
- Protected route behavior works.
- Loading and empty states render.
- Status badges render correct labels.
- Layout is usable on desktop and tablet widths.
