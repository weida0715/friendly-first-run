## RFC-013: End-to-End Website Completion and Acceptance Testing

### Status

Proposed

### Summary

Validate the complete website as a full FYP2 deliverable. This issue tests Normal User, Moderator, Administrator, Public Hub, export, and acceptance workflows.

### Problem Statement

After all modules are implemented, the full system must be tested as an integrated website. The final website should match the prototype and support all proposed BEE workflows.

### Goals

- Stabilize integration issues.
- Test full Normal User workflow.
- Test Moderator workflow.
- Test Administrator workflow.
- Test Public Hub and export workflow.
- Add final regression suite.
- Add requirements traceability evidence.

### Non-Goals

- No major new feature development.
- Only integration fixes and final tests.

### Design Classes Covered

All major Boundary, Control, and Entity classes.

### Implementation Plan

#### Stage 1: Resolve integration issues across complete workflow

Resolve integration issues discovered after combining frontend, backend, database, queue, worker, market data, and experiment execution modules. Fix route mismatches between prototype-aligned frontend pages and Flask APIs. Fix status synchronization between ExperimentWizardView, QueueService, JobDetailView, worker updates, ExperimentDetailView, and ModelsRankingsView. Fix Blueprint approval visibility between BlueprintApprovalController, ExperimentWizardController, and PublicHubController. Fix BTCUSDT 1m refresh timing so DefaultExperimentExecutor refreshes data before loading cached candles on every run. Fix favorite model and favorite Blueprint UI state updates while preserving Boundary-Control-Entity separation.

Refs: ExperimentWizardView, QueueService, DefaultExperimentExecutor, BlueprintApprovalController, PublicHubController

#### Stage 2: Add normal user experiment workflow tests

Add end-to-end tests for the Normal User workflow. Verify a user can register through RegistrationView, log in through LoginView, reach DashboardView, create a Blueprint through BlueprintWizardView, request approval from BlueprintDetailView, create an Experiment through ExperimentWizardView using an approved Blueprint, trigger BTCUSDT 1m data refresh, enqueue a job through QueueService, execute through DefaultExperimentExecutor, generate Model and Model metric columns and ExperimentLog metric records, view ExperimentDetailView, inspect ModelDetailView, sort results in ModelsRankingsView, and favorite a Model through ModelsLibraryView.

Tests: Normal User E2E workflow

#### Stage 3: Add moderator blueprint governance workflow tests

Add end-to-end tests for the Moderator workflow. Verify Moderator login, BlueprintModerationView access, pending Blueprint listing, Blueprint approval, Blueprint rejection, Blueprint disapproval, automatic public visibility changes, approved Blueprint availability in ExperimentWizardView, rejected Blueprint owner remediation through VersioningService, and restricted access to Administrator-only UserController actions.

Tests: Moderator E2E workflow

#### Stage 4: Add administrator user and system workflow tests

Add end-to-end tests for the Administrator workflow. Verify Administrator login, UserManagementView access, user search, create Normal User, enable user, disable user, update role, reset password, delete user, SystemManagementView access, active job queue visibility, session timeout setting visibility, and queue limit setting visibility. Confirm Administrator actions update User records correctly and do not corrupt related Blueprint, Experiment, Model, FavoriteModel, or FavoriteBlueprint records.

Tests: Administrator E2E workflow

#### Stage 5: Add public hub discovery and export workflow tests

Add end-to-end tests for Public Hub and export workflows. Verify authenticated users can browse PublicHubView tabs for enabled Users, successful Experiments, eligible Models, and approved Blueprints. Verify private artifacts are hidden, disabled users are hidden, failed experiments are hidden, draft and pending Blueprints are hidden, search and filters work, and navigation to UserProfileView, ExperimentDetailView, ModelDetailView, and BlueprintDetailView respects access rules. Verify completed Experiment logs and metrics can be exported through LogsDownloadController.

Tests: Public Hub E2E and export workflow

#### Stage 6: Add requirements traceability and final regression suite

Add final acceptance and regression test suite for the completed BEE website. Map tests to requirements for authentication, user management, Blueprint governance, experiment configuration, BTCUSDT 1m market data refresh and cache storage, split-first execution, parameter permutation generation, model artifact creation, model ranking, Public Hub discovery, security, usability, and deployment readiness. Verify all module tests, integration tests, security tests, and end-to-end tests pass together. Generate requirements traceability evidence for the FYP2 Testing and Evaluation chapter.

Tests: acceptance testing, requirements traceability, final regression suite
