## RFC-001: Project Foundation and Layered Architecture Skeleton

### Status

Proposed

### Summary

Create the initial BEE monorepo structure and align the codebase with the final FYP1 Chapter 5 design model. This issue establishes the frontend, backend, domain, repository, infrastructure, worker, controller, service, validator, strategy, and executor folders before feature development begins.

This RFC also defines the project versioning, dependency locking, environment configuration, and reproducible setup requirements so the frontend and backend can be installed, tested, and demonstrated consistently.

### Problem Statement

Without a clear project structure, later implementation may mix UI, business logic, persistence, and infrastructure code. This would weaken the reportâ€™s design-to-implementation traceability and make the final system harder to test. BEE must therefore begin with a structure that reflects the Boundary, Control, and Entity layers from the final design model.

Additionally, without dependency locking and environment versioning, the project may behave differently across machines or during demonstration. Since BEE depends on a Next.js frontend, Flask backend, Python services, SQLAlchemy persistence, Binance Connector, and worker dependencies, the project must define reproducible versions from the beginning.

### Goals

- Initialize the Next.js 15 frontend.
- Initialize the Flask backend.
- Create a project tree that reflects the FYP1 design classes.
- Prepare folders for views, controllers, services, validators, repositories, strategies, executors, infrastructure, workers, domain models, and tests.
- Configure project versioning, dependency locking, and reproducible environment setup for both frontend and backend.
- Add documented environment variables through `.env.example`.
- Add the first frontend-to-backend health-check integration.
- Add startup, dependency, environment, and health-check tests.

### Non-Goals

- No real authentication implementation yet.
- No database schema implementation yet.
- No experiment execution logic yet.
- No BTCUSDT kline retrieval yet.
- No production deployment yet.
- No actual secret values should be committed.

### Project Versioning and Dependency Management

The project shall use explicit versioning and dependency locking to ensure the implementation can be reproduced during development, testing, demonstration, and FYP2 evaluation.

#### Application Versioning

The project shall maintain a root `VERSION` file using semantic versioning.

Initial version:

```text
0.1.0
```

Versioning convention:

```text
MAJOR.MINOR.PATCH
```

* `MAJOR`: breaking architecture, API, or database changes.
* `MINOR`: new module or major feature added.
* `PATCH`: bug fixes, test fixes, UI corrections, or documentation updates.

Suggested milestone versions:

```text
0.1.0 - Project foundation
0.2.0 - Database and domain schema complete
0.3.0 - Authentication and RBAC complete
0.4.0 - Blueprint workflow complete
0.5.0 - BTCUSDT 1m data retrieval and cache complete
0.6.0 - Experiment configuration and queue complete
0.7.0 - Split-first execution pipeline complete
0.8.0 - Model ranking and Public Hub complete
0.9.0 - Security and deployment hardening complete
1.0.0 - Final FYP2 submission version
```

The frontend shall use `package.json` and `package-lock.json` to define and lock dependencies.

The frontend shall also include `.nvmrc` to document the Node.js version.

Example:

```text
20
```

#### Backend Dependency Management

The backend shall use `requirements.txt`, `requirements-dev.txt`, and optionally `pyproject.toml`.

Required backend dependencies include:

```text
Flask
Flask-Login
Flask-WTF
SQLAlchemy
Alembic
python-dotenv
Werkzeug
pandas
numpy
scikit-learn
binance-connector
redis
rq
pytest
```

The backend shall include `.python-version` to document the Python version.

Example:

```text
3.11
```

#### Environment Configuration

The project shall include `.env.example` to document required environment variables without committing real secrets.

Example:

```env
FLASK_ENV=development
SECRET_KEY=change-me
DATABASE_URL=postgresql://bee_user:bee_password@localhost:5432/bee
REDIS_URL=redis://localhost:6379/0
NEXT_PUBLIC_API_BASE_URL=http://localhost:5000/api
BINANCE_BASE_URL=https://api.binance.com
SESSION_TIMEOUT_MINUTES=1440
```

Actual `.env` files must be ignored by Git.

#### Dependency Reproducibility

The project shall commit lock files where applicable:

```text
frontend/package-lock.json
backend/requirements.txt
backend/requirements-dev.txt
```

This ensures the same dependency versions can be installed during development, testing, demonstration, and report evaluation.

### Implementation Plan

#### Stage 1: Scaffold layered bee monorepo structure

Create the initial BEE monorepo and organize it according to the FYP1 Chapter 5 final design model. Add frontend and backend root folders. In the frontend, create app routes, reusable component folders, API helper folders, and view files representing Boundary Layer classes. In the backend, create folders for controllers, services, validators, repositories, strategies, executors, infrastructure, domain models, value objects, workers, and tests. Add placeholder files for Control Layer classes and service classes.

Refs: FYP1 Chapter 5 Boundary, Control, and Entity design"

#### Stage 2: Initialize nextjs shadcn tailwind base app

Initialize the frontend application using Next.js 15, React, Tailwind CSS, and Shadcn/ui. Add app router layout, global styles, Shadcn/ui configuration, component path aliases, and placeholder route pages for landing, login, register, dashboard, experiments, blueprints, models, public hub, admin users, system management, documentation, and job detail pages. Add BaseView and WizardView placeholder components so later pages can share navigation, page heading, loading, empty-state, and validation display patterns.

Refs: BaseView, WizardView, LandingPageView, RegistrationView, LoginView, DashboardView

#### Stage 3: Initialize flask app factory and route registry

Initialize the Flask backend with an application factory, configuration loader, JSON response helpers, route registry, and environment-specific settings. Add placeholder route modules for authentication, users, blueprints, experiments, models, public hub, documentation, logs, jobs, and system management. Prepare dependency wiring points so future controllers can receive repositories and services through dependency injection rather than importing concrete infrastructure directly.

Refs: AuthenticationController, UserController, BlueprintController, ExperimentController, ModelController, PublicHubController, JobController

#### Stage 4: Add versioning dependency locks and environment examples

Add reproducible project versioning and dependency management for the BEE monorepo. Add root VERSION file using semantic versioning, CHANGELOG.md for implementation milestones, .env.example for documented environment variables, and .gitignore for generated files and secrets. Add frontend package.json and package-lock.json for Next.js 15, React, Tailwind CSS, Shadcn/ui-related packages, lucide-react, class-variance-authority, tailwind-merge, and lightweight-charts. Add frontend .nvmrc for Node.js version documentation. Add backend pyproject.toml, requirements.txt, requirements-dev.txt, and .python-version for Flask, Flask-Login, Flask-WTF, SQLAlchemy, Alembic, Binance Connector, Redis/RQ, pandas, numpy, scikit-learn, and pytest dependencies. This ensures that the project can be reproduced during development, testing, demonstration, and FYP2 evaluation.

Refs: project foundation, dependency management, reproducible environment setup

#### Stage 5: Add health check endpoint and frontend api client

Add the first working frontend-backend integration point. Implement a Flask health-check endpoint that returns service status, application name, version, and environment. Add a frontend API client wrapper with base URL configuration, JSON parsing, error normalization, and health-check request support. Render the backend health status in a lightweight frontend diagnostic component.

Refs: LandingPageView, DashboardView, DashboardController

#### Stage 6: Add startup dependency environment and health tests

Add foundation tests for the initial BEE project setup. Verify the Flask app factory creates an application instance, required environment variables are documented in .env.example, VERSION exists and follows semantic versioning, frontend package-lock.json exists, backend requirements files exist, the health-check endpoint returns ok=true, the frontend API client can parse the health response, and the base frontend layout renders without crashing. These tests ensure that the project foundation is reproducible and ready for future feature modules.

Tests: app factory, dependency files, version file, environment config, health endpoint, frontend api client, base layout
