### 7.5 System Testing

System testing validates the behaviour of the Bitcoin Experimental Engine (BEE) as an integrated application rather than as isolated units or pairwise integrations. The purpose of this section is to confirm that the implemented modules work together from the perspective of real user workflows, role-based system access, application navigation, persistence, and Blueprint governance.

The current system testing scope is based on the project state completed through RFC-003, RFC-004, and RFC-005. Therefore, this section validates the implemented system as an authenticated, role-governed, prototype-aligned research platform foundation with strict ERD persistence and governed Blueprint workflows. Full market-data ingestion, asynchronous queue execution, split-first model training, generated model rankings, completed Public Hub publication, and export workflows are not claimed as completed system-tested features because they belong to later RFC stages.

#### 7.5.9 System Testing Summary

System testing confirms that the current BEE implementation operates coherently as a role-governed web application foundation. Guests can register and log in, authenticated users can access protected application pages, staff and administrator functions are restricted by role, and server-managed sessions maintain current-user identity across protected routes. User-management workflows also operate correctly across frontend views, backend controllers, access-control services, repositories, and database persistence.

The prototype-matching frontend has been validated as an integrated user interface shell. Major routes render their expected views, authenticated pages share a consistent layout, role-aware navigation behaves correctly, and shared UI components such as loading states, error states, empty states, status badges, dialogs, and wizard layouts function consistently across the application.

The governed Blueprint workflow has also been validated at system level. Users can create Draft Blueprints, receive validation feedback, view owned and favorited Blueprints, inspect Blueprint details, favorite and unfavorite accessible Blueprints, request approval, and rely on immutable versioning rules when editing reviewed artifacts. Moderators and Administrators can perform approval, rejection, and disapproval actions, while normal users are blocked from moderation workflows. Approved Blueprints are exposed to the experiment wizard selection boundary, confirming the intended handoff point between the completed Blueprint workflow and later experiment-configuration work.

Overall, system testing shows that the current implementation is stable for the RFC-003 to RFC-005 scope. The system is not yet a complete experiment execution engine; instead, it is a tested foundation for authenticated access, strict ERD persistence, prototype-aligned UI interaction, user governance, and Blueprint governance. Later system testing will be required once BTCUSDT ingestion, queue-worker execution, split-first model generation, model rankings, Public Hub publication, export workflows, and security hardening are completed.
