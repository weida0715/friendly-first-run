### 7.3 Unit Testing

Unit testing was conducted to verify the correctness of individual modules implemented in the current Bitcoin Experimental Engine (BEE) scope. The unit tests focus on isolated behaviours in domain entities, value objects, services, validators, repositories, frontend components, route guards, and workflow views. This section follows the same module separation used in Section 6.2 so that the tested units can be traced directly to the implemented features.

The current implementation scope covers the project foundation, strict ERD persistence, authentication and role-based access control, prototype-matching frontend components, and the Blueprint workflow delivered through RFC-003, RFC-004, and RFC-005. Therefore, the unit testing evidence in this section focuses on those implemented modules. Features planned for later RFCs, such as full queue-worker execution, Binance data ingestion, split-first model training, model ranking computation, and complete Public Hub publication, are not treated as completed unit-test evidence in this section.

#### 7.3.4 Unit Testing Summary

The unit testing completed for the current implementation confirms that the RFC-003 to RFC-005 features behave correctly at module level. Authentication tests verify secure registration, password hashing, login, logout, current-user lookup, disabled-account rejection, and server-managed session handling. Access-control tests verify the role hierarchy among User, Moderator, and Admin accounts and confirm that staff user-management actions are constrained by role.

Frontend unit tests confirm that the prototype-matching boundary layer is stable. Major route views render successfully, authenticated and role-restricted guards behave as expected, shared layout components render consistently, status badges display correct labels, and common loading, error, empty, dialog, and wizard states work as reusable UI units.

Blueprint unit tests confirm the main governed artifact workflow delivered by RFC-005. Valid Blueprints can be created as Drafts, invalid payloads are rejected, owned and favorited library views work, detail visibility is role-aware, favorite and unfavorite actions persist correctly, versioning rules preserve reviewed artifacts, approval requests transition Blueprints into Pending state, and Moderator/Admin actions transition Blueprints through Approved, Rejected, and Disapproved states.

Database and repository unit tests confirm that the strict ERD persistence layer remains reliable. The tests verify PascalCase table and column naming, enum values, database constraints, repository mapping behaviour, Unit of Work transaction handling, and BTCUSDT kline timestamp upsert behaviour. These tests support the integrity of the implemented authentication and Blueprint workflows because both rely on consistent domain-to-database mapping.

Overall, the unit test results show that the implemented BEE foundation is stable for the current scope. The results do not claim completion of later RFC functionality such as full Binance ingestion, asynchronous experiment execution, split-first model generation, model ranking computation, Public Hub publication, export workflows, or full CSRF hardening. Those areas remain future testing targets for later implementation stages.
