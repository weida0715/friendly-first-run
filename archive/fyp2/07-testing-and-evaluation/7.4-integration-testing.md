### 7.4 Integration Testing

Integration testing was conducted to verify that independently tested modules work correctly when connected across application boundaries. While unit testing in Section 7.3 validates individual classes, services, repositories, and UI components, integration testing validates the interaction between frontend views, API clients, Flask controllers, backend services, repositories, database persistence, authentication context, and role-based access-control decisions.

The integration testing scope follows the current implementation state of BEE and the completed RFC-003 to RFC-005 modules. Therefore, the tests focus on authentication and session flows, role-based user management, prototype-aligned frontend-to-backend boundaries, strict ERD-backed persistence, and the governed Blueprint workflow. Later features such as full Binance ingestion, asynchronous queue-worker execution, split-first model training, generated model rankings, complete Public Hub publication, and CSV export are not treated as completed integration-tested workflows in this section.

#### 7.4.5 Integration Testing Summary

The integration tests demonstrate that the implemented BEE modules work correctly across their main boundaries. Authentication and session tests confirm that frontend login and registration views interact correctly with Flask authentication endpoints, password hashing, user repositories, and server-managed sessions. Role-based tests confirm that frontend guards and backend access-control services apply the same User, Moderator, and Admin hierarchy.

User-management integration tests confirm that staff-facing frontend views, API client functions, backend controllers, access-control checks, password services, and repositories work together to list, create, update, disable, reset, and delete users according to role restrictions. These tests validate the RFC-003 multi-user foundation required for later Blueprint governance and experiment workflows.

Frontend integration tests confirm that the RFC-004 app shell, navigation model, route views, reusable components, shared states, status badges, dialogs, and wizard layout work together to provide a consistent prototype-aligned user interface. These tests are important because the frontend includes many module boundaries that are reused by the implemented Blueprint workflow and later planned experiment/model/Public Hub workflows.

Blueprint integration tests confirm the most complete domain workflow in the current system. The tests validate the full connection from Blueprint wizard submission to backend validation, authenticated owner context, Unit of Work persistence, library listing, detail retrieval, favorite actions, immutable versioning, approval request, moderation transitions, and approved Blueprint option exposure for the experiment wizard. This provides evidence that BEE can already govern reusable research artifacts through role-controlled workflows.

Overall, integration testing confirms that the current RFC-003 to RFC-005 implementation is coherent across frontend, backend, service, repository, and database boundaries. It also identifies a clear future testing boundary for RFC-006 onward, where market-data retrieval, experiment execution, queue processing, model generation, ranking, Public Hub publication, exports, and deployment hardening will require additional integration tests after those features are completed.
