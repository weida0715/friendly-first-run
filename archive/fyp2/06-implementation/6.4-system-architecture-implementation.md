### 6.4 System Architecture Implementation

The implemented Bitcoin Experimental Engine (BEE) follows the layered architecture direction established in the design chapter and the project foundation defined in RFC-001. RFC-001 required the codebase to begin with a monorepo structure that separates the frontend Boundary layer from backend controllers, services, validators, repositories, infrastructure adapters, workers, strategies, executors, and domain objects. This structure was selected so that user-interface concerns, request handling, business rules, persistence logic, and infrastructure dependencies would not be mixed together as the system grows.

At the current implementation stage, this layered structure has been realized through a `frontend/` workspace for the Next.js web application and a `backend/` workspace for the Flask API and Python application layers. The current implementation covers the project foundation, strict ERD-aligned persistence, authentication and role-based access control, prototype-aligned frontend shell components, and the governed Blueprint workflow. Later modules such as full market-data refresh, queue-worker execution, model-ranking data logic, export workflows, and complete Public Hub discovery are represented by prepared boundaries or placeholders, but they are not treated as fully completed architectural flows in this section.

#### 6.4.1 Frontend Presentation Layer

The frontend presentation layer is implemented in the `frontend/` workspace using Next.js 15, React, TypeScript, Tailwind CSS, and shadcn-style reusable components. This layer corresponds to the Boundary/View layer from the design model. It is responsible for rendering pages, collecting user input, showing loading and error states, and calling backend APIs through a controlled client module. It does not directly access database or backend infrastructure code.

The Next.js App Router structure is implemented under `frontend/app/`. Route files map URL paths to Boundary-layer views under `frontend/views/`. For example, the landing page is routed through `frontend/app/page.tsx`, authenticated pages such as dashboard, experiments, Blueprints, models, Public Hub, documentation, jobs, profile, user management, and system management are represented through route folders under `frontend/app/`, and each route imports a corresponding view component. This follows the RFC-001 frontend tree where route files act as thin composition boundaries and the main page behavior is placed in view components.

The global frontend layout is implemented in `frontend/app/layout.tsx`. It wraps all pages with `AuthProvider` from `frontend/lib/auth/AuthProvider.tsx` and `AppShell` from `frontend/components/layout/AppShell.tsx`. `AuthProvider` loads the current authenticated user through the backend current-user endpoint and exposes authentication state to the frontend. `AppShell` provides the shared visual frame for authenticated application pages, including the top bar, sidebar navigation, breadcrumbs, and content region. Authentication pages such as login and registration are excluded from the shell so they can use standalone layouts.

Reusable layout components are implemented in `frontend/components/layout/`, including `AppShell`, `TopBar`, `SidebarNav`, `Breadcrumbs`, `PageShell`, and `PageHeader`. These components enforce consistent structure across the application and support the prototype-matching frontend requirement. Navigation visibility is controlled through `frontend/lib/routes/nav.ts`, which maps route items to minimum roles and filters available navigation links according to the authenticated user role. This keeps navigation presentation aligned with the same role hierarchy used by the backend.

The frontend also implements reusable Boundary-layer components for common interface behavior. `frontend/views/BaseView.tsx` provides a shared page foundation with loading, error, and content rendering behavior. `frontend/views/WizardView.tsx` provides the reusable multi-step structure used by wizard-style workflows such as Blueprint authoring and experiment configuration. Shared form, table, state, status, and dialog components are implemented under `frontend/components/forms/`, `frontend/components/tables/`, `frontend/components/states/`, `frontend/components/status/`, and `frontend/components/ui/`. These components reduce duplicated UI logic and keep view files focused on user workflows.

Authentication and authorization controls also appear in the presentation layer. `frontend/lib/auth/guards.tsx` defines `RequireAuth` and `RequireRole`, which protect authenticated and staff/admin pages from unauthenticated or unauthorized access. These guards do not replace backend authorization, but they improve user flow by preventing restricted pages from rendering before server-side access decisions are requested. The implemented frontend therefore follows a layered boundary pattern: route files compose views, views render user workflows, shared components provide reusable interface structure, and API calls are centralized through `frontend/lib/api/client.ts` and `frontend/lib/api/endpoints.ts`.

#### 6.4.2 Backend Presentation Layer

The backend presentation layer is implemented using Flask controllers under `backend/app/controllers/`. This layer receives HTTP requests, validates request payloads at the boundary where appropriate, resolves authentication context, calls services or repositories, and returns normalized JSON responses. It represents the Control-facing HTTP boundary from the design model and preserves RFC-001's requirement for a central route registry and separated controller modules.

The backend application is created in `backend/app/__init__.py` through the `create_app()` factory. The factory loads environment-specific configuration from `backend/app/config.py`, applies API CORS handling for configured origins, warns when the temporary in-memory session backend is used outside testing, and delegates endpoint registration to `backend/app/routes.py`. The route registry registers all controller blueprints under the configured `/api` prefix, including system health, authentication, users, experiments, Blueprints, Blueprint approval, Blueprint library, models, Public Hub, documentation, jobs, and logs.

Response formatting is centralized in `backend/app/responses.py`. Successful responses, error responses, and validation error responses follow a consistent JSON shape. This ensures that frontend API client code can handle responses predictably and that controllers do not duplicate response-building logic. The health endpoint exposed by the system controller also supports the frontend-to-backend integration required by RFC-001.

The implemented controllers reflect both the initial RFC-001 skeleton and the later completed modules. `authentication_controller.py` implements registration, login, current-user lookup, and logout flows. `user_controller.py` implements user profile and staff user-management endpoints. `blueprint_controller.py`, `blueprints_library_controller.py`, and `blueprint_approval_controller.py` implement Blueprint creation, detail retrieval, favorites, owned/favorited listings, approval requests, and moderation actions. Other controller files, such as experiment, model, Public Hub, documentation, job, and logs controllers, preserve the architectural boundaries required by the design model and provide prepared locations for later RFC implementation.

This organization keeps HTTP concerns separate from persistence and domain rules. Controllers do not define database schemas or directly own SQLAlchemy table mappings. Instead, they use services, validators, repositories, and the Unit of Work pattern where required. As a result, the backend presentation layer remains a request/response boundary rather than becoming a mixed business-and-data layer.

#### 6.4.3 Business Logic Layer

The business logic layer is implemented mainly under `backend/app/services/` and `backend/app/validators/`, with prepared extension points under `backend/app/executors/` and `backend/app/strategies/`. This layer holds application rules that should not be embedded inside route handlers or frontend components.

Authentication and authorization logic is represented through `PasswordService`, `SessionService`, and `AccessControlService`. `PasswordService` encapsulates password hashing and verification so that controllers do not manipulate password hashes directly. `SessionService` manages server-side session creation, lookup, timeout validation, and destruction. `AccessControlService` resolves authentication context and implements role-based checks for User, Moderator, and Admin access. It is used by controllers to enforce staff and ownership-related access rules.

Blueprint governance logic is implemented through `BlueprintValidator` and `VersioningService`. `BlueprintValidator` checks Blueprint payloads for required metadata, architecture configuration, and indicator configuration before persistence. `VersioningService` enforces the immutable versioning rule: draft Blueprints that have never been submitted may be edited in place, while previously submitted or reviewed Blueprints are preserved and revised through versioned draft copies. This keeps reproducibility and auditability concerns outside the frontend and outside raw repository operations.

The current codebase also includes validators, services, executors, and strategies for later experimentation workflows. `ExperimentValidator`, `QueueService`, `MarketDataService`, `ExperimentExecutor`, `DefaultExperimentExecutor`, `BlueprintExecutorFactory`, `ReferenceArchitectureExecutor`, and related strategy classes exist as architectural boundaries for the experiment configuration, market-data, queue-worker, and split-first execution modules. Their presence follows RFC-001's requirement to prepare folders for validators, strategies, executors, workers, and infrastructure before feature development. However, the current completed scope should be understood as the foundation and Blueprint/auth workflow rather than the full execution engine.

This business logic organization supports the design chapter's layered architecture and preserves a clear direction for future split-first execution. Validators enforce input and domain constraints, services coordinate application policies, strategies prepare interchangeable execution and cancellation behavior, and executors provide the location for fixed-order experiment execution. This separation is important because BEE's long-term reproducibility goal depends on preventing user-interface code or database access code from controlling the experiment sequence directly.

#### 6.4.4 Data Access Layer

The data access layer is implemented through SQLAlchemy ORM mappings, repository classes, Alembic migrations, and a Unit of Work transaction boundary. This layer is located mainly under `backend/app/infrastructure/database/`, `backend/app/infrastructure/database/orm/`, `backend/app/repositories/`, and `backend/alembic/`.

The current database implementation follows the strict ERD-aligned persistence foundation delivered after RFC-001. Physical schema naming follows the Chapter 4 ERD and RFC-002 rules rather than generic snake_case table names. The ERD-approved persistent entities are `User`, `Blueprint`, `Experiment`, `Model`, `ExperimentLog`, `FavoriteModel`, `FavoriteBlueprint`, and `BTCUSDTKline`. Their ORM mappings are placed under `backend/app/infrastructure/database/orm/`, while domain entities are represented separately under `backend/app/domain/models/`.

Repository classes encapsulate database access for each aggregate or persistence concern. Examples include `UserRepository`, `BlueprintRepository`, `ExperimentRepository`, `ModelRepository`, `ExperimentLogRepository`, `FavoriteModelRepository`, `FavoriteBlueprintRepository`, and `MarketDataRepository`. These repositories convert between ORM rows and domain objects, expose query methods such as lookup, listing, add, update, delete, favorite checks, and market-data range retrieval, and prevent controllers from depending on raw SQLAlchemy queries.

Transaction management is provided through `backend/app/repositories/unit_of_work.py`. The Unit of Work creates a session scope, exposes repository instances, and handles commit, rollback, and close behavior. This supports the data integrity requirement that related writes should be grouped into explicit transaction boundaries. The database session factory is configured in `backend/app/infrastructure/database/session.py`, which resolves a PostgreSQL `DATABASE_URL`, creates the SQLAlchemy engine, and binds the session factory lazily.

Alembic migrations under `backend/alembic/versions/` define the physical schema. The baseline RFC-002 migration creates the strict ERD schema, constraints, enum values, keys, and relationships. A later migration extends Blueprint approval state support for the disapproved moderation state. Repository and database tests under `backend/tests/repositories/` and `backend/tests/database/` verify that naming, constraints, enum behavior, repository behavior, and transaction handling remain aligned with the persistence design.

This layer keeps persistence logic independent from controller and frontend code. Business rules call repositories through defined methods, while infrastructure-specific details such as engine configuration, ORM table names, and migrations remain inside the infrastructure and repository boundary.

#### 6.4.5 Infrastructure Layer

The infrastructure layer contains adapters and technical configuration that support the application but should not define user workflows directly. In the current project, infrastructure is located under `backend/app/infrastructure/` and includes database, Binance, Redis, session, and CSRF-related folders. Some of these infrastructure boundaries are fully used by the current implementation, while others are prepared for later RFCs.

The most active infrastructure component is the database infrastructure. `backend/app/infrastructure/database/session.py` configures the SQLAlchemy engine and session factory and enforces the PostgreSQL runtime requirement. `backend/app/infrastructure/database/base.py` and the ORM package support declarative mappings for the ERD-backed persistence layer. Alembic configuration in `backend/alembic/env.py` and `backend/alembic.ini` supports migration execution.

Configuration infrastructure is implemented in `backend/app/config.py` and environment variables documented through `.env.example` and the repository README. The configuration layer defines application name, version, API prefix, secret key, database URL, Redis URL, Binance base URL, session timeout, cookie settings, session backend, and CORS origins. The Flask app factory reads this configuration and applies CORS headers for API requests from configured frontend origins.

Session infrastructure currently uses a server-managed session service with an in-memory backend by default. The application factory logs warnings when this temporary mode is used outside testing, especially in multi-worker deployments. This makes the current limitation explicit and preserves the intended direction toward centralized session infrastructure such as Redis.

The Binance, Redis, CSRF, worker, executor, and strategy directories exist as prepared boundaries from RFC-001 and later roadmap items. Dependencies for Binance Connector, Redis, RQ, pandas, NumPy, and scikit-learn are present in backend dependency files, and frontend charting dependency support is present through `lightweight-charts`. These boundaries allow future market-data retrieval, queue-worker processing, CSRF hardening, and experiment execution modules to be integrated without changing the overall architecture. At the current stage, they should be described as prepared or scaffolded infrastructure boundaries unless the specific feature flow has been completed.

#### 6.4.6 Overall Project Structure

The overall project structure reflects RFC-001's goal of preserving traceability between the FYP design model and the implementation. The root repository contains versioning, documentation, environment, frontend, backend, and archive materials. The frontend workspace contains route files, Boundary-layer views, reusable components, API/auth helpers, validators, route metadata, and frontend tests. The backend workspace contains the Flask application factory, route registry, controllers, services, validators, repositories, domain models, infrastructure adapters, strategies, executors, workers, migrations, and backend tests.
