### 6.7 Backend Implementation

The backend implementation of the Bitcoin Experimental Engine (BEE) is located in the `backend/` workspace and is implemented using Flask, Python 3.11, SQLAlchemy, Alembic, and pytest. The backend represents the Control, Service, Domain, Repository, and Infrastructure layers described in the design model. Its main responsibility is to receive API requests from the Next.js frontend, validate request payloads, enforce authentication and authorization rules, coordinate domain services, persist data through repositories, and return normalized JSON responses.

The backend implementation at the current stage corresponds mainly to RFC-003, RFC-004, and RFC-005 on top of the RFC-001 and RFC-002 foundation. RFC-003 implemented authentication, session handling, role-based access control, and staff user management. RFC-004 required backend route boundaries to support the prototype frontend pages. RFC-005 implemented the governed Blueprint workflow, including validation, draft persistence, library/detail retrieval, favorites, immutable versioning, approval requests, and moderation actions. Later modules such as full experiment creation, market-data refresh, queue-worker execution, model ranking, Public Hub discovery, CSV exports, and CSRF hardening are represented by prepared boundaries or placeholder controllers, but they are not treated as fully completed backend workflows in this section.

#### 6.7.1 Flask API Structure

The backend uses a Flask application factory pattern. The application factory is implemented in `backend/app/__init__.py`. It creates the Flask application instance, loads environment-aware configuration from `backend/app/config.py`, applies API CORS handling, warns when the temporary in-memory session backend is used outside testing, and registers all API route blueprints through `backend/app/routes.py`.

The backend route registry keeps HTTP endpoint composition in one place. Each controller module defines a Flask `Blueprint`, and the route registry attaches these blueprints to the configured `/api` prefix. This design keeps the app factory lightweight and preserves a clear separation between application setup and feature-specific request handling.

The main backend structure is summarized below.

```text
backend/
  app/
    __init__.py              # Flask application factory
    config.py                # Environment-aware configuration
    routes.py                # Central API route registry
    responses.py             # Normalized JSON response helpers
    controllers/             # HTTP/API controllers
    services/                # Application services and policy logic
    validators/              # Request/domain validation rules
    repositories/            # Repository classes and UnitOfWork
    domain/                  # Domain models and value objects
    infrastructure/          # Database, Binance, Redis, sessions, CSRF boundaries
    executors/               # Experiment execution boundaries for later RFCs
    strategies/              # Strategy/factory boundaries for later RFCs
    workers/                 # Worker boundary for later RFCs
  alembic/                   # Database migration environment
  tests/                     # pytest backend tests
```

The currently registered API groups include system health, authentication, users, experiments, Blueprints, Blueprint approval, Blueprint library, models, Public Hub, documentation, jobs, and logs. Not all registered route groups contain full production workflows yet. Some exist to preserve the Chapter 5 design boundary and to support RFC-004 prototype routes until later RFCs complete their backend behavior.

Pseudocode for the Flask application factory:

```text

    load configuration object using config_name or FLASK_ENV

        log warning about temporary in-memory session backend
    END IF

    return app
END FUNCTION
```

#### 6.7.2 Configuration and Database Session Setup

Backend configuration is implemented in `backend/app/config.py`. It defines shared application settings such as application name, version, API prefix, environment name, secret key, database URL, Redis URL, Binance base URL, session timeout, session cookie settings, session backend mode, CORS origins, and testing/debug flags. The application version is read from the root `VERSION` file so that health responses and documentation remain traceable to the repository version.

Database session setup is implemented in `backend/app/infrastructure/database/session.py`. The current backend is PostgreSQL-first. At runtime, `DATABASE_URL` is required and must use a PostgreSQL scheme. The SQLAlchemy engine is created lazily when a repository or Unit of Work first requests database access. This avoids accidental database initialization during import and ensures that runtime persistence uses the configured PostgreSQL database.

Pseudocode for configuration selection:

```text
    ELSE IF selected is production

    ELSE
        return DevelopmentConfig
    END IF
```
```text
END FUNCTION
```

#### 6.7.3 Authentication and Session Management

Authentication and session management were implemented as part of RFC-003. The authentication controller is implemented in `backend/app/controllers/authentication_controller.py`. It provides endpoints for registration, login, current-user lookup, and logout. Password hashing and verification are encapsulated in `backend/app/services/password_service.py`, while session creation, lookup, timeout validation, and destruction are handled by `backend/app/services/session_service.py`.

The backend stores only password hashes, not plaintext passwords. During registration, the backend validates name, username, email, and password fields, checks duplicate username and email through `UserRepository`, hashes the password, and persists a new `User` domain entity through the Unit of Work. During login, the backend retrieves the user by email, verifies the password hash, rejects disabled accounts, creates a server-managed session, and sets the session identifier in an HTTP-only cookie.

Pseudocode for registration payload validation:

```text
FUNCTION validate_registration_payload(payload)
    name = trimmed payload.name
    username = trimmed payload.username
    email = trimmed lowercase payload.email
    password = payload.password

    IF name is empty
        return invalid "Name is required"
    END IF

    IF username is empty
        return invalid "Username is required"
    END IF

    IF username length is outside accepted range
        return invalid "Username length is invalid"
    END IF

    IF username is not lowercase alphanumeric
        return invalid "Username format is invalid"
    END IF

    IF email is empty OR email does not contain @
        return invalid "A valid email is required"
    END IF

    IF password is shorter than minimum length
        return invalid "Password is too short"
    END IF

    return valid
END FUNCTION
```

Pseudocode for registration endpoint:

```text
ENDPOINT POST /api/auth/register
    payload = read JSON request body

    IF payload is not object
        return error INVALID_REGISTRATION
    END IF

    IF validation is invalid
        return error INVALID_REGISTRATION with message
    END IF

    normalize name, username, email, password

    TRY
        WITH UnitOfWork as uow
            IF uow.users.get_by_username(username) exists

            END IF

                return conflict EMAIL_EXISTS
            END IF

            now = current UTC datetime
            user = User(
                user_id = null,
                username = username,
                email = email,
                password_hash = hash_password(password),
                name = name,
                role = "User",
                status = "Enabled",
                created_at = now,
                updated_at = now
            )

            createdUser = uow.users.add(user)
        END WITH
    CATCH database integrity error
        return conflict USER_CONFLICT
    END TRY

    return success response with created user summary and status 201
END ENDPOINT
```

Pseudocode for session service:

```text
CLASS SessionService
    SHARED sessionStore = map sessionId to SessionRecord

    METHOD create_server_session(user_id, role)
        purge expired sessions
        sessionId = generate secure random token
        now = current UTC datetime
        expiresAt = now + configured timeout duration
        record = SessionRecord(sessionId, user_id, role, createdAt, expiresAt)
        store record by sessionId
        return record
    END METHOD

    METHOD get_server_session(session_id)
        purge expired sessions
        record = sessionStore[session_id]

        IF record does not exist
            return null
        END IF

        IF record is expired
            remove record
            return null
        END IF

        return record
    END METHOD

    METHOD destroy_server_session(session_id)
        remove sessionId from sessionStore if present
    END METHOD

    METHOD resolve_session_from_cookie(request, cookie_name)
        sessionId = request cookie value by cookie_name
        IF sessionId does not exist
            return null
        END IF
        return get_server_session(sessionId)
    END METHOD
END CLASS
```

#### 6.7.4 Access Control and Role-Based Authorization

Role-based access control was implemented as part of RFC-003 and is centralized in `backend/app/services/access_control_service.py`. This service resolves the authenticated user context from the server-managed session cookie, loads the corresponding user through the repository layer, and exposes helper methods for authentication, role checks, ownership checks, profile access, user-management permissions, and role assignment rules.

The role hierarchy used by the backend is `User`, `Moderator`, and `Admin`. A Moderator is considered staff and can access moderation and limited user-management functions. An Admin has the highest role and can perform full user-management and system-level actions. Controllers use `AccessControlService` to avoid duplicating role logic across endpoints.

Pseudocode for authenticated context resolution:

```text
CLASS AccessControlService
    METHOD get_authenticated_context(request)
        sessionRecord = sessionService.resolve_session_from_cookie(request, cookie_name)

        IF sessionRecord is null
            return null
        END IF

        WITH UnitOfWork as uow
            user = uow.users.get_by_id(sessionRecord.user_id)
        END WITH

        IF user does not exist
            return null
        END IF

        IF user.status is not Enabled
            return null
        END IF

        return AuthContext(
            user_id = user.user_id,
            username = user.username,
            email = user.email,
            name = user.name,
            role = user.role,
            status = user.status
        )
    END METHOD
END CLASS
```

Pseudocode for role ranking:

```text
FUNCTION normalize_role(role)
    IF role matches Admin or Administrator
        return Admin
    ELSE IF role matches Moderator or Mod
        return Moderator
    ELSE
        return User
    END IF
END FUNCTION

FUNCTION role_rank(role)
    normalizedRole = normalize_role(role)

    IF normalizedRole is User
        return 1
    ELSE IF normalizedRole is Moderator
        return 2
    ELSE IF normalizedRole is Admin
        return 3
    END IF
END FUNCTION
```

Pseudocode for access-control checks:

```text
FUNCTION is_staff(actor)
    return role_rank(actor.role) >= role_rank(Moderator)
END FUNCTION

FUNCTION is_moderator(actor)
    return normalize_role(actor.role) is Moderator
END FUNCTION

FUNCTION is_admin(actor)
    return normalize_role(actor.role) is Admin
END FUNCTION

FUNCTION is_owner(actor, target_user_id)
    return actor.user_id equals target_user_id
END FUNCTION

FUNCTION can_access_profile(actor, target_user_id)
    return is_owner(actor, target_user_id) OR is_staff(actor)
END FUNCTION
```

Pseudocode for user-management permissions:

```text
FUNCTION can_manage_user(actor, targetUser)
    IF actor is Admin
        return true
    END IF

    IF actor is Moderator AND targetUser.role is User
        return true
    END IF

    return false
END FUNCTION

FUNCTION can_assign_role(actor, new_role)
    IF actor is not Admin
        return false
    END IF

    return new_role is one of User, Moderator, Admin
END FUNCTION
```


#### 6.7.6 Blueprint Validation and Draft Persistence

Blueprint validation and draft persistence were implemented as part of RFC-005. The validation rules are implemented in `backend/app/validators/blueprint_validator.py`. The validator checks whether the submitted Blueprint payload contains valid metadata, architecture configuration, and indicator configuration. Validation failures are returned through the normalized validation error response format so that the frontend can display field-specific errors in the Blueprint wizard.

Blueprint creation is handled by `backend/app/controllers/blueprint_controller.py`. The create endpoint requires an authenticated user, validates the payload, constructs a `Blueprint` domain entity, and persists it as a `Draft` owned by the current user through the Unit of Work.

Pseudocode for Blueprint validator:

```text
CLASS BlueprintValidator
    METHOD validate(payload)
        errors = empty map

        IF payload is not object
            errors.payload = "Blueprint payload must be an object"
            return ValidationResult false with errors
        END IF

        metadata = payload.metadata
        IF metadata is missing OR metadata.name is empty
            errors["metadata.name"] = "Blueprint name is required"
        END IF

        architecture = payload.architecture
        IF architecture is missing
            errors["architecture"] = "Architecture is required"
        ELSE
            IF architecture.reference is empty
                errors["architecture.reference"] = "Reference architecture is required"
            END IF
            IF architecture.settings is not object
                errors["architecture.settings"] = "Architecture settings must be an object"
            END IF
        END IF

        indicators = payload.indicators
        IF indicators is missing OR indicators.selected is not list
            errors["indicators.selected"] = "Selected indicators must be provided"
        END IF

        IF errors is not empty
            return ValidationResult false with errors
        END IF

        return ValidationResult true with no errors
    END METHOD
END CLASS
```

Pseudocode for creating a draft Blueprint:

```text
ENDPOINT POST /api/blueprints/
    payload = read JSON request body

    IF payload is not object
        return invalid payload error
    END IF

    accessControl = build access-control service
    context = accessControl.get_authenticated_context(request)

    IF context is null
        return unauthenticated response
    END IF

    validation = BlueprintValidator.validate(payload)
    IF validation is not ok
        return validation_error_response(validation.errors)
    END IF

    metadata = payload.metadata
    now = current UTC datetime

    draft = Blueprint(
        blueprint_id = null,
        user_id = context.user_id,
        name = metadata.name,
        description = metadata.description,
        indicators = payload.indicators,
        features = empty object,
        architecture = payload.architecture,
        approval_state = "Draft",
        submitted_at = null,
        version = 1,
        parent_id = null,
        created_at = now,
        updated_at = now
    )

    WITH UnitOfWork as uow
        created = uow.blueprints.add(draft)
    END WITH

    return success response with created Blueprint id, version, approvalState, detailPath and status 201
END ENDPOINT
```

#### 6.7.7 Blueprint Detail, Library, and Favorite Workflow

The Blueprint detail and library workflows were implemented as part of RFC-005. The backend allows users to retrieve accessible Blueprint details, list Blueprints they own, list Blueprints they have favorited, and favorite or unfavorite accessible Blueprints. Visibility is enforced by controller-level checks using the current authenticated context and role information.

A Blueprint is accessible when the viewer is the owner, the Blueprint is approved, or the viewer is a staff user. Draft and disapproved Blueprints are not publicly visible to unrelated users. The detail response includes metadata, indicators, architecture, approval state, version, lineage, owner information, and viewer context.

Pseudocode for Blueprint visibility check:

```text
FUNCTION can_access_blueprint(blueprint, user_id, is_staff)
    isOwner = user_id is not null AND user_id equals blueprint.user_id
    isPublic = blueprint.approval_state equals Approved

    return isOwner OR isPublic OR is_staff
END FUNCTION
```

```text
            return not found BLUEPRINT_NOT_FOUND
        END IF

        isOwner = context exists AND context.user_id equals blueprint.user_id

        IF can_access_blueprint(blueprint, context.user_id, isStaff) is false
            return forbidden "Blueprint is not visible"
        END IF

        owner = uow.users.get_by_id(blueprint.user_id)
        parent = uow.blueprints.get_by_id(blueprint.parent_id) if parent_id exists
        children = uow.blueprints.list_by_parent(blueprint.blueprint_id)
        isFavorited = context exists AND uow.favorite_blueprints.exists(context.user_id, blueprint.blueprint_id)
    END WITH

    return success response containing:
        Blueprint metadata
        indicators
        architecture
        approvalState
        version
        lineage parent and children
        owner summary
        viewer flags: isAuthenticated, isOwner, isStaff, role, isFavorited
```

#### 6.7.8 Blueprint Versioning Service

The immutable Blueprint versioning rule was implemented in `backend/app/services/versioning_service.py` as part of RFC-005. This rule supports reproducibility by preventing previously submitted or reviewed Blueprints from being silently mutated. A Blueprint that remains a never-submitted draft can be edited in place. However, when an owner edits a Blueprint that has already been submitted, approved, rejected, or disapproved, the service creates a new draft copy with an incremented version number and a parent reference to the original Blueprint.

This behavior preserves the original artifact for auditability while still allowing users to revise their research specifications. The service is used by the Blueprint update endpoint.

Pseudocode for saving Blueprint edits:

```text
CLASS VersioningService
    METHOD save_blueprint_edit(existingBlueprint, payload, actor_user_id)
        IF existingBlueprint.user_id does not equal actor_user_id
            raise PermissionError
        END IF

        metadata = payload.metadata
        now = current UTC datetime

        patch = {
            name: metadata.name,
            description: metadata.description,
            indicators: payload.indicators,
            features: existingBlueprint.features or empty object,
            architecture: payload.architecture,
            updated_at: now
        }

        IF existingBlueprint.approval_state is Draft AND existingBlueprint.submitted_at is null
            updated = repository.update(existingBlueprint.blueprint_id, patch)
            return updated
        END IF

        children = repository.list_by_parent(existingBlueprint.blueprint_id)
        highestVersion = maximum version among existingBlueprint and children
        newVersion = highestVersion + 1

        versionedDraft = Blueprint(
            blueprint_id = null,
            user_id = existingBlueprint.user_id,
            name = patch.name,
            description = patch.description,
            indicators = patch.indicators,
            features = patch.features,
            architecture = patch.architecture,
            approval_state = "Draft",
            submitted_at = null,
            version = newVersion,
            parent_id = existingBlueprint.blueprint_id,
            created_at = now,
            updated_at = now
        )

        created = repository.add(versionedDraft)
        return created
    END METHOD
END CLASS
```

Pseudocode for Blueprint update endpoint:


#### 6.7.9 Blueprint Approval and Moderation

#### 6.7.12 Repository and Unit of Work Usage

The backend uses repository classes to keep database access separate from controllers and services. Repositories are instantiated through `UnitOfWork`, which opens a SQLAlchemy session, constructs repository instances, commits on successful completion, rolls back on exceptions, and closes the session.

The repositories most directly used by RFC-003 to RFC-005 include `UserRepository`, `BlueprintRepository`, and `FavoriteBlueprintRepository`. Additional repositories for experiments, models, experiment logs, favorite models, and market data are implemented as part of the strict ERD persistence foundation and prepared for later workflows.

Pseudocode for Unit of Work:

```text
CLASS UnitOfWork
    METHOD enter_context()
        initialize database engine if needed
        create SQLAlchemy session
        create UserRepository(session)
        create BlueprintRepository(session)
        create ExperimentRepository(session)
        create ModelRepository(session)
        create ExperimentLogRepository(session)
        create FavoriteModelRepository(session)
        create FavoriteBlueprintRepository(session)
        create MarketDataRepository(session)
        return this UnitOfWork
    END METHOD

    METHOD exit_context(exception)
        IF exception does not exist
            TRY
                commit session
            CATCH error
                rollback session
                re-raise error
            END TRY
        ELSE
            rollback session
        END IF

        close session
    END METHOD

    METHOD commit()
        commit active session
    END METHOD

    METHOD rollback()
        rollback active session
    END METHOD
END CLASS
```
