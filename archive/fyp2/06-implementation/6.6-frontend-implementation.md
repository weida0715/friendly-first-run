### 6.6 Frontend Implementation

The frontend implementation of the Bitcoin Experimental Engine (BEE) was developed in the `frontend/` workspace using Next.js 15, React 19, TypeScript, Tailwind CSS, and shadcn-style reusable components. This frontend represents the Boundary layer described in the design model. Its responsibility is to provide browser-based access to authentication, role-governed navigation, dashboard pages, user management, Blueprint authoring, Blueprint library browsing, moderation, and prototype-aligned placeholders for later experiment, model, Public Hub, documentation, job, and system workflows. The frontend does not access the database directly. Instead, it communicates with the Flask backend through centralized API helper functions under `frontend/lib/api/`.

The implemented frontend scope corresponds mainly to RFC-003, RFC-004, and RFC-005. RFC-003 introduced the authentication, session, route protection, and user-management user interfaces. RFC-004 established the prototype-matching application shell, navigation, route structure, shared page foundations, and reusable UI components. RFC-005 extended the frontend with Blueprint authoring, Blueprint library/detail views, favorite handling, immutable versioning entry points, and moderation workflows. Features from later RFCs, such as full experiment execution, BTCUSDT chart visualization, model ranking data logic, CSV exports, and complete Public Hub discovery, are represented by routes or prototype-aligned views but are not treated as fully completed frontend workflows at this implementation stage.

#### 6.6.1 Application Routing

The frontend uses the Next.js App Router. Route files are located under `frontend/app/`, while most page-level interface logic is implemented in `frontend/views/`. This keeps route files thin and preserves the Boundary-layer separation between URL composition and reusable view implementation. The global layout is defined in `frontend/app/layout.tsx`, which wraps the application with `AuthProvider` and `AppShell`. The `AuthProvider` loads the current authenticated user, while the `AppShell` provides the shared top bar, sidebar, breadcrumb area, and main content region for authenticated application pages.

#### 6.6.2 Authentication Provider and Route Guard Implementation

The authentication frontend implementation was delivered as part of RFC-003. The central authentication state is implemented in `frontend/lib/auth/AuthProvider.tsx`. This provider calls the backend current-user endpoint through the API client, stores the authenticated user in React state, exposes loading and authenticated status, and provides `refreshUser()` and `logout()` operations to child components.

Route-level authorization is implemented through `frontend/lib/auth/guards.tsx`. The `RequireAuth` component protects pages that require an authenticated user. If the user is not authenticated after the loading state completes, the guard redirects the browser to the login page and preserves the intended destination through the `next` query parameter. The `RequireRole` component protects staff and administrator pages by checking the authenticated user's role against the required minimum role. The frontend role hierarchy follows the same logical ordering used by the backend access-control service: `User`, `Moderator`, and `Admin`.

The frontend authentication flow is used by the login, registration, dashboard, profile, user-management, system, Blueprint, experiment, model, Public Hub, documentation, and job routes. However, these frontend guards are not the only security boundary. The backend still performs authorization checks through controllers and `AccessControlService`, while the frontend guards improve navigation behavior and prevent unauthorized interfaces from rendering prematurely.

Pseudocode for `AuthProvider`:

```text
COMPONENT AuthProvider(children)
    STATE user = null
    STATE isLoading = true

    FUNCTION refreshUser()
        TRY
            response = call getCurrentUser API
            user = response.data.user OR null
        CATCH error
            IF error status is 401
                user = null
            ELSE
                user = null
            END IF
        END TRY
    END FUNCTION

    FUNCTION logout()
        TRY
            call logout API
        FINALLY
            user = null
        END TRY
    END FUNCTION

    ON component mounted
        call refreshUser()
        set isLoading = false after refresh finishes
    END ON

    PROVIDE auth context containing:
        user
        isLoading
        isAuthenticated = user is not null
        refreshUser
        logout

    render children
END COMPONENT
```

Pseudocode for `RequireAuth`:

```text
COMPONENT RequireAuth(children)
    auth = read AuthProvider context
    pathname = read current browser path

    IF auth is not loading AND auth is not authenticated
        redirect to /login?next=currentPath
    END IF

    IF auth is loading OR auth is not authenticated
        render nothing
    ELSE
        render children
    END IF
END COMPONENT
```

Pseudocode for `RequireRole`:

```text
COMPONENT RequireRole(minimumRole, fallbackTo, children)
    auth = read AuthProvider context
    userRole = normalize auth.user.role
    authorized = roleRank(userRole) >= roleRank(minimumRole)

    IF auth is not loading AND auth is not authenticated
        redirect to /login
    END IF

    IF auth is not loading AND auth is authenticated AND authorized is false
        redirect to fallbackTo
    END IF

    IF auth is loading OR auth is not authenticated OR authorized is false
        render nothing
    ELSE
        render children
    END IF
END COMPONENT
```

#### 6.6.3 Navigation and Application Shell

The prototype-matching application shell was implemented under RFC-004. The shell components are located in `frontend/components/layout/` and include `AppShell`, `TopBar`, `SidebarNav`, `Breadcrumbs`, `PageShell`, `PageHeader`, and `Navbar`.

The navigation model is defined in `frontend/lib/routes/nav.ts`. Each navigation item defines a label, route, icon, section, and minimum role. The sidebar filters navigation items based on whether the user is authenticated and whether the user's role satisfies the item's minimum role. Normal users can access core research pages such as dashboard, experiments, Blueprints, models, Public Hub, documentation, and profile. Moderators additionally gain access to staff functions such as user management, moderation, and jobs. Administrators gain access to administrator-only system management.

This navigation design supports the role-based interface requirements from RFC-003 while preserving the prototype layout requirements from RFC-004. The application shell also supports responsive behavior through a mobile sidebar pattern and desktop sidebar layout, allowing the interface to remain usable across desktop and tablet-class screens as required by the non-functional usability requirements.

Pseudocode for `AppShell`:

```text
COMPONENT AppShell(children)

        render TopBar with mobile menu button
        render layout row
            render SidebarNav for desktop navigation
            render main content area
                render Breadcrumbs
                render page content children
            end main content area
        end layout row

        IF mobileSidebarOpen is true
            render MobileSidebar containing SidebarNav
        END IF
    end container
END COMPONENT
```

Pseudocode for `TopBar`:

```text
COMPONENT TopBar(onOpenMobileNav)
```

Pseudocode for `SidebarNav`:

```text

FUNCTION getVisibleNavItems(isAuthenticated, role)
```


#### 6.6.5 Registration and Login Views

The registration and login interfaces were implemented as part of RFC-003. `RegistrationView` provides the user-facing registration form and connects to the backend registration endpoint through `registerUser()` in `frontend/lib/api/client.ts`. The view collects name, username, email, password, and confirmation fields, applies client-side validation, submits the payload to the backend, and displays validation or conflict errors returned by the API.

`LoginView` provides the authentication form for existing users. It submits email and password credentials through `loginUser()`, handles invalid credential errors, and refreshes the frontend authentication context after a successful login. Once authenticated, the user is redirected into the protected application area. The login and registration validators are separated under `frontend/lib/validators/`, keeping form validation logic outside the page layout components.

Together, these views implement the user-facing part of account creation and authentication. They work with backend session creation, password verification, and current-user lookup so that authenticated pages can determine whether a user is logged in and what role the user holds.

Pseudocode for `RegistrationView`:

```text
COMPONENT RegistrationView()
    STATE form = { name, username, email, password, confirmPassword }
    STATE fieldErrors = empty map
    STATE submitError = null
    STATE isSubmitting = false

    FUNCTION handleSubmit()
        clear errors
        validationResult = validate registration form

        IF validationResult is invalid
            set fieldErrors
            STOP
        END IF

        TRY
            set isSubmitting = true
            call registerUser API with name, username, email, password
            redirect to login or dashboard depending on flow
        CATCH ApiClientError
            map backend error to user-facing message
            set submitError or fieldErrors
        FINALLY
            set isSubmitting = false
        END TRY
    END FUNCTION

    render registration form
        render name input
        render username input
        render email input
        render password input
        render confirm password input
        render field validation messages
        render submit error if present
        render submit button
    end form
END COMPONENT
```

Pseudocode for registration validation:

```text
FUNCTION validateRegistration(form)
    errors = empty map

    IF name is empty
        errors.name = "Name is required"
    END IF

    IF username is invalid length or format
        errors.username = "Username format is invalid"
    END IF

    IF email is invalid
        errors.email = "Valid email is required"
    END IF

    IF password is too short
        errors.password = "Password is too short"
    END IF

    IF password does not equal confirmPassword
        errors.confirmPassword = "Passwords do not match"
    END IF

    return validation result with errors
END FUNCTION
```

Pseudocode for `LoginView`:

```text
COMPONENT LoginView()
    STATE form = { email, password }
    STATE fieldErrors = empty map
    STATE submitError = null
    STATE isSubmitting = false
    auth = read AuthProvider context
    nextRoute = read next query parameter or /dashboard

    FUNCTION handleSubmit()
        clear errors
        validationResult = validate login form

        IF validationResult is invalid
            set fieldErrors
            STOP
        END IF

        TRY
            set isSubmitting = true
            call loginUser API with email and password
            call auth.refreshUser()
            redirect to nextRoute
        CATCH ApiClientError
            set submitError from backend error message
        FINALLY
            set isSubmitting = false
        END TRY
    END FUNCTION

    render login form
        render email input
        render password input
        render validation errors
        render submit error if present
        render submit button
        render link to registration page
    end form
END COMPONENT
```

#### 6.6.6 User Management and Profile Views

The staff user-management interface was implemented to support RFC-003 user administration workflows. `UserManagementView` provides a role-governed administrative interface for viewing users, filtering user records, creating managed users, enabling or disabling accounts, resetting passwords, updating roles, and deleting users where permitted by the actor's role.

The interface uses shared table, badge, form, and dialog components. User roles and account statuses are displayed through status badge components so that staff users can quickly identify whether an account is a normal user, moderator, administrator, enabled account, disabled account, or pending account. Actions are constrained in the UI according to the role hierarchy. For example, moderator-level users can perform staff-level operations permitted by the backend, while administrator-only actions are only exposed to administrators.

`UserProfileView` provides the authenticated user's profile boundary. It is connected to protected routing and current-user state. This supports the general profile and account visibility requirements while preserving backend authority over which profile information can be accessed.

Pseudocode for `UserManagementView`:

```text
COMPONENT UserManagementView()
    STATE users = empty list
    STATE query = empty string
    STATE roleFilter = empty
    STATE statusFilter = empty
    STATE page = 1
    STATE loading = true
    STATE error = null
    auth = read current user

    FUNCTION loadUsers()
        TRY
            set loading = true
            response = call listUsers API with query, roleFilter, statusFilter, page
            users = response.data.items
        CATCH error
            set error message
        FINALLY
            set loading = false
        END TRY
    END FUNCTION

    FUNCTION createUser(form)
        call createManagedUser API
        reload users
    END FUNCTION

    FUNCTION updateStatus(userId, status)
        call updateManagedUserStatus API
        reload users
    END FUNCTION

    FUNCTION resetPassword(userId, password)
        call resetManagedUserPassword API
        reload users or show success message
    END FUNCTION

    FUNCTION updateRole(userId, role)
        IF current user is not Admin
            do not show role action
        ELSE
            call updateManagedUserRole API
            reload users
        END IF
    END FUNCTION

    FUNCTION deleteUser(userId)
        IF current user is not Admin
            do not show delete action
        ELSE
            call deleteManagedUser API
            reload users
        END IF
    END FUNCTION

    ON filters or page changed
        loadUsers()
    END ON

    render BaseView with user-management title
        render TableToolbar with search and filters
        render create-user action
        render DataTable
            FOR each user
                render username, email, name
                render UserRoleBadge
                render UserStatusBadge
                render allowed action buttons
            END FOR
        end DataTable
    end BaseView
END COMPONENT
```

Pseudocode for `UserProfileView`:

```text
COMPONENT UserProfileView()
    auth = read AuthProvider context

    IF auth is loading
        render LoadingState
    ELSE IF auth.user is null
        render nothing or redirect through RequireAuth
    ELSE
        render BaseView titled "Profile"
            render user name
            render username
            render email
            render UserRoleBadge
            render UserStatusBadge
        end BaseView
    END IF
END COMPONENT
```

#### 6.6.7 Dashboard and Module Landing Pages

#### 6.6.8 Blueprint Wizard Implementation

The Blueprint wizard was implemented under RFC-005. It is available through `/blueprints/new` and is rendered by `BlueprintWizardView`. The wizard allows authenticated users to create reusable experiment pipeline specifications as Blueprint records. The implemented wizard follows a multi-step authoring flow that includes basic metadata, reference architecture selection, indicator selection, and review before submission.

The Blueprint wizard manages local form state for the Blueprint name, description, selected reference architecture, selected indicators, and parameter values. It performs client-side validation before allowing users to progress through the steps and also handles backend validation errors returned by the Blueprint creation API. When the user submits a valid Blueprint, the frontend calls `createBlueprint()` from the API client. The backend then validates and persists the Blueprint as a Draft record owned by the authenticated user.

The current Blueprint wizard supports RFC-005's implemented scope: draft creation, validation error surfacing, duplicate-submit protection, and navigation to the created Blueprint's detail page after successful submission. More advanced future behavior, such as compiling executable experiment manifests for full experiment execution, remains part of later experiment and execution RFCs.

Pseudocode for `BlueprintWizardView`:

```text
COMPONENT BlueprintWizardView(mode, sourceBlueprintId)
    STATE currentStep = 1
    STATE config = {
        name,
        description,
        category,
        tags,
        indicators,
        referenceModelId,
        referenceModelParams
    }
    STATE fieldErrors = empty map
    STATE stepError = null
    STATE submitError = null
    STATE isSubmitting = false

    FUNCTION validateStep(step)
        errors = empty map

        IF step is Basics
            IF name is empty
                errors.name = "Blueprint name is required"
            END IF
        END IF

        IF step is Reference Architecture
            IF referenceModelId is empty
                errors.referenceModelId = "Select a reference architecture"
            END IF
            FOR each architecture parameter
                validate numeric comma-separated values
            END FOR
        END IF

        IF step is Indicators
            FOR each selected indicator parameter
                validate required numeric comma-separated values
            END FOR
        END IF

        set fieldErrors = errors
        return errors is empty
    END FUNCTION

    FUNCTION handleNext()
        IF validateStep(currentStep) is true
            increment currentStep
        END IF
    END FUNCTION

    FUNCTION handleBack()
        decrement currentStep if currentStep > 1
    END FUNCTION

    FUNCTION toggleIndicator(indicatorId)
        IF indicator is already selected
            remove it from config.indicators
        ELSE
            add indicator with default parameters
        END IF
    END FUNCTION

    FUNCTION selectReferenceModel(modelId)
        set referenceModelId
        copy default architecture parameters into config
    END FUNCTION

    FUNCTION handleCreate()
        IF already submitting
            STOP
        END IF

        build parameter_ranges from architecture and indicator parameter values

        TRY
            set isSubmitting = true
            response = call createBlueprint API with metadata, indicators, architecture, parameter_ranges
            redirect to response.data.blueprint.detailPath OR /blueprints
        CATCH ApiClientError
            IF backend validation errors exist
                map backend errors to local fieldErrors
                set submitError
            ELSE
                set submitError to error message
            END IF
        FINALLY
            set isSubmitting = false
        END TRY
    END FUNCTION

    render wizard page
        render step navigation
        IF currentStep is 1
            render basics form
        ELSE IF currentStep is 2
            render reference architecture selection
        ELSE IF currentStep is 3
            render indicator selection and parameter form
        ELSE IF currentStep is 4
            render review summary
        END IF
        render back/next/create controls
    end wizard page
END COMPONENT
```

#### 6.6.10 Blueprint Moderation View

The Blueprint moderation interface was implemented to support moderator and administrator review workflows from RFC-005. It is routed through `/blueprints/moderation` and rendered by `BlueprintModerationView`. Access to this page is protected through role-based frontend guards so that normal users cannot access staff moderation interfaces.

The moderation view loads pending Blueprint submissions from the backend moderation queue endpoint. It displays submitted Blueprints with relevant metadata such as name, version, approval state, submitted timestamp, and updated timestamp. Staff users can approve, reject, or disapprove Blueprints through moderation action buttons. These actions call backend endpoints that enforce staff authorization and update the Blueprint approval state.

This interface implements the frontend portion of Blueprint governance. Approved Blueprints become available for experiment selection, rejected Blueprints remain visible to their owners for revision, and disapproved Blueprints are removed from public visibility while preserving the audit trail required by the versioning and moderation workflow.

Pseudocode for `BlueprintModerationView`:

```text
COMPONENT BlueprintModerationView()
    STATE queueItems = empty list
    STATE loading = true
    STATE error = null
    STATE actionLoadingId = null

    FUNCTION loadModerationQueue()
        TRY
            set loading = true
            response = call getBlueprintModerationQueue API
            queueItems = response.data.items
        CATCH error
            set error message
        FINALLY
            set loading = false
        END TRY
    END FUNCTION

    FUNCTION moderateBlueprint(blueprintId, action)
        TRY
            set actionLoadingId = blueprintId
            IF action is approve
                call approveBlueprint API
            ELSE IF action is reject
                call rejectBlueprint API
            ELSE IF action is disapprove
                call disapproveBlueprint API
            END IF
            reload moderation queue
        CATCH error
            display action error
        FINALLY
            set actionLoadingId = null
        END TRY
    END FUNCTION

    ON component mounted
        loadModerationQueue()
    END ON

    render BaseView titled "Blueprint Moderation"
        IF loading
            render LoadingState
        ELSE IF error exists
            render ErrorState
        ELSE IF queueItems is empty
            render EmptyState
        ELSE
            render table of pending Blueprints
                FOR each item
                    render name, version, approval state, submitted date, updated date
                    render approve button
                    render reject button
                    render disapprove button
                END FOR
            end table
        END IF
    end BaseView
END COMPONENT
```

#### 6.6.12 API Client and Frontend-Backend Integration

Frontend-to-backend communication is centralized in `frontend/lib/api/client.ts` and `frontend/lib/api/endpoints.ts`. The API client defines helper functions for GET, POST, PATCH, and DELETE requests, attaches `credentials: 'include'` so that the browser sends the server-managed session cookie, parses JSON responses, and raises `ApiClientError` when backend responses are unsuccessful.

The endpoint configuration uses a same-origin proxy path by default. Browser-side calls prefer `/api/backend`, which is rewritten by `frontend/next.config.ts` to the Flask backend API origin. This design avoids common local CORS problems while still supporting deployment configuration through environment variables such as `NEXT_PUBLIC_API_BASE_URL` and `BACKEND_API_ORIGIN`.

The implemented API client currently includes functions for backend health checks, user registration, login, logout, current-user lookup, user listing and staff user-management actions, Blueprint creation, Blueprint favorite and unfavorite actions, Blueprint approval requests, Blueprint moderation actions, Blueprint moderation queue retrieval, and approved Blueprint options for experiment selection.

This centralized API boundary prevents view components from constructing raw URLs or duplicating fetch error handling. It also keeps frontend integration consistent with the normalized JSON response pattern used by the backend.

Pseudocode for API endpoint building:

```text
FUNCTION buildApiUrl(path)
    IF path does not start with slash
        normalizedPath = slash + path
    ELSE
        normalizedPath = path
    END IF

    return API_BASE_URL + normalizedPath
END FUNCTION
```

Pseudocode for API response parsing:

```text
FUNCTION parseJsonResponse(response)
    IF response content type is JSON
        payload = parse response JSON
    ELSE
        payload = null
    END IF

    IF response status is not successful
        message = payload.error.message OR default status message
        THROW ApiClientError(message, response.status, payload)
    END IF

    return payload
END FUNCTION
```

Pseudocode for generic API helpers:

```text
FUNCTION apiGet(path)
```
