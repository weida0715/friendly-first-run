### 6.5 Database Implementation

The database implementation establishes the persistence foundation for the Bitcoin Experimental Engine (BEE). It follows RFC-002, which requires the implemented schema to match the Chapter 4 third-normal-form ERD exactly. This means that the physical database does not use generic snake_case table names such as `users`, `blueprints`, or `btcusdt_klines`. Instead, the implemented schema preserves the ERD naming convention by using singular PascalCase table names, PascalCase column names, and ERD-defined key names such as `UserID`, `BlueprintID`, `ExperimentID`, `ModelID`, `ExperimentLogID`, and `Timestamp`.

The runtime database target is PostgreSQL. The database engine and session factory are configured in `backend/app/infrastructure/database/session.py`, where `DATABASE_URL` is required and validated to use a PostgreSQL scheme. This PostgreSQL-first implementation prevents the application from silently running against a different database backend during normal execution and keeps development behaviour aligned with deployment expectations.

#### 6.5.1 Database Schema Overview

The implemented schema consists only of the ERD-approved persistent tables. RFC-002 explicitly prevents the creation of additional persistence tables such as `JobRecord`, `ServerSession`, `ExperimentEvaluationMetrics`, or `ExperimentConfusionMetrics` unless the ERD is formally revised. Therefore, job state, centralized session storage, or extended evaluation tables are not part of the current strict ERD database schema. Model-level metrics are stored in the `Model` table, while row-level or structured evaluation details are stored through the `ExperimentLog.Metrics` JSON field.

| Physical Table | Primary Key | Purpose |
| --- | --- | --- |
| `User` | `UserID` | Stores account identity, login credential hash, display name, role, status, and timestamps. |
| `Blueprint` | `BlueprintID` | Stores reusable experiment pipeline specifications, approval state, version number, and lineage through `ParentID`. |
| `Experiment` | `ExperimentID` | Stores experiment configuration, selected Blueprint, date range, split ratios, parameter overrides, status, progress, and completion metadata. |
| `Model` | `ModelID` | Stores generated model artifacts for an experiment, including parameter configuration and summary metric columns. |
| `ExperimentLog` | `ExperimentLogID` | Stores timestamped model output records, signals, predictions, and flexible metrics JSON linked to an experiment and model. |
| `FavoriteModel` | Composite `UserID`, `ModelID` | Stores the many-to-many favorite relationship between users and models. |
| `FavoriteBlueprint` | Composite `UserID`, `BlueprintID` | Stores the many-to-many favorite relationship between users and Blueprints. |
| `BTCUSDTKline` | `Timestamp` | Stores BTCUSDT OHLCV market candles with one record per timestamp. |

The baseline migration implementing these tables is located at `backend/alembic/versions/20260503_0001_rfc002_strict_erd.py`. A later migration, `20260503_0002_rfc005_add_disapproved_approval_state.py`, extends the Blueprint approval-state vocabulary to support the moderation workflow delivered after the baseline persistence foundation.

The backend separates domain entities from ORM mappings. Domain classes are implemented under `backend/app/domain/models/`, while physical database mappings are implemented under `backend/app/infrastructure/database/orm/`. The ORM layer preserves exact ERD table and column names, while Python code can still use Pythonic attribute names internally.

#### 6.5.3 Migration and Schema Implementation

The baseline schema is implemented through Alembic. The migration creates all ERD-approved tables in dependency order, defines primary keys and foreign keys, applies enum values, and adds the unique and check constraints required by RFC-002.

The migration creates the `User` table first because several tables depend on `UserID`. It then creates `Blueprint`, followed by `Experiment`, `Model`, `ExperimentLog`, the two favorite relationship tables, and `BTCUSDTKline`. The downgrade operation drops the same tables in reverse dependency order to avoid foreign-key conflicts.

The core migration pattern is summarized below.

```python
def upgrade():
    create_table("User", columns=[UserID, Username, Email, PasswordHash, ...])
    create_table("Blueprint", foreign_keys=[UserID, ParentID], constraints=[unique_owner_name_version])
    create_table("Experiment", foreign_keys=[UserID, BlueprintID], constraints=[split_sum, min_val_test])
    create_table("Model", foreign_keys=[ExperimentID])
    create_table("ExperimentLog", foreign_keys=[ExperimentID, ModelID])
    create_table("FavoriteModel", composite_primary_key=[UserID, ModelID])
    create_table("FavoriteBlueprint", composite_primary_key=[UserID, BlueprintID])
    create_table("BTCUSDTKline", primary_key=Timestamp)
```

The schema uses non-native SQLAlchemy enum definitions for values such as user roles, account statuses, Blueprint approval states, experiment intervals, and experiment statuses. This keeps the allowed values explicit in the migration and supports database-level verification through automated tests. The implemented role values are `User`, `Moderator`, and `Admin`; the implemented user status values are `Enabled`, `Disabled`, and `Pending`; and experiment status values include `Queued`, `Running`, `Completed`, `Failed`, and `Cancelled`.

The database session setup is initialized lazily. When a repository or Unit of Work requests a session, the application resolves `DATABASE_URL`, validates that it is PostgreSQL-based, creates the SQLAlchemy engine, and binds `SessionLocal` to that engine.

```python
def get_engine():
    if engine_exists:
        return engine

    database_url = resolve_database_url()
    engine = create_engine(database_url, future=True)
    SessionLocal.configure(bind=engine)
    return engine
```

This design avoids accidental database initialization during import and ensures that runtime persistence always uses the configured PostgreSQL database.

#### 6.5.4 Repository and Unit of Work Implementation

Repository classes are implemented under `backend/app/repositories/`. Each repository encapsulates persistence operations for a specific ERD-approved table or relationship. Controllers and services therefore do not need to construct SQLAlchemy queries directly. Instead, they call repository methods such as `add()`, `get_by_id()`, `list_by_user()`, `update()`, `remove()`, `exists()`, or `list_range()` depending on the persistence concern.

Below is the pseudocode for each implemented backend repository in `backend/app/repositories` so far.

1. `UserRepository`

```text
CLASS UserRepository
    CONSTANT DEFAULT_SEARCH_LIMIT = 20
    CONSTANT MIN_SEARCH_LIMIT = 1
    CONSTANT MAX_SEARCH_LIMIT = 100
    CONSTANT MIN_OFFSET = 0

    METHOD initialize(session)
        store database session

    METHOD convert_orm_to_domain(user_row)
        extract role value from enum if needed
        extract status value from enum if needed
        return User domain object using ORM row fields

    METHOD add(user)
        convert user role to UserRole enum
        convert user status to UserStatus enum
        create UserORM row from User domain object
        add row to session
        flush session to generate database values
        return converted domain User

    METHOD get_by_id(user_id)
        find UserORM by primary key
        if found, return domain User
        otherwise return null

    METHOD get_by_username(username)
        query UserORM where Username equals username
        if found, return domain User
        otherwise return null

    METHOD get_by_email(email)
        query UserORM where Email equals email
        if found, return domain User
        otherwise return null

    METHOD list_all()
        query all users ordered by UserID
        convert each row to domain User
        return user list

    METHOD search_users(query, role, status, limit, offset)
        start query from UserORM

        if query is provided
            normalize query to lowercase wildcard pattern
            filter where username OR email OR name matches query

        if role is provided
            filter by Role

        if status is provided
            filter by Status

        clamp limit between MIN_SEARCH_LIMIT and MAX_SEARCH_LIMIT
        clamp offset to at least MIN_OFFSET

        execute query ordered by UserID with limit and offset
        convert rows to domain Users
        return users

    METHOD count_users(query, role, status)
        start count query for UserID

        if query is provided
            normalize query to lowercase wildcard pattern
            filter where username OR email OR name matches query

        if role is provided
            filter by Role

        if status is provided
            filter by Status

        execute count query
        return count or 0

    METHOD update_status(user_id, status)
        find user by ID
        if not found, return null
        convert status to UserStatus enum
        update row Status
        flush session
        return updated domain User

    METHOD update_role(user_id, role)
        find user by ID
        if not found, return null
        convert role to UserRole enum
        update row Role
        flush session
        return updated domain User

    METHOD update_password_hash(user_id, password_hash)
        find user by ID
        if not found, return null
        update PasswordHash
        flush session
        return updated domain User

    METHOD delete_by_id(user_id)
        find user by ID
        if not found, return false
        delete user row
        flush session
        return true
END CLASS
```

Transaction management is implemented through `UnitOfWork`. The Unit of Work opens a SQLAlchemy session, constructs repository instances using that session, and commits if the operation completes successfully. If an exception occurs, it rolls the transaction back and then closes the session.

Pseudocode:

`UnitOfWork`

```text
CLASS UnitOfWork
    METHOD initialize()
        set session to null
        set users repository to null
        set blueprints repository to null
        set experiments repository to null
        set models repository to null
        set experiment_logs repository to null
        set favorite_models repository to null
        set favorite_blueprints repository to null
        set market_data repository to null

    METHOD enter_context()
        initialize database engine if needed
        create new database session

        create UserRepository using session
        create BlueprintRepository using session
        create ExperimentRepository using session
        create ModelRepository using session
        create ExperimentLogRepository using session
        create FavoriteModelRepository using session
        create FavoriteBlueprintRepository using session
        create MarketDataRepository using session

        return this UnitOfWork instance

    METHOD exit_context(exception_type, exception, traceback)
        if session does not exist
            return

        try
            if no exception occurred
                try
                    commit session
                catch commit error
                    rollback session
                    re-raise error
            else
                rollback session
        finally
            close session

    METHOD commit()
        if session does not exist
            raise RuntimeError
        commit session

    METHOD rollback()
        if session does not exist
            raise RuntimeError
        rollback session
END CLASS
```

This transaction boundary is important for database consistency. It allows related persistence operations to be treated as a single unit and prevents partial writes when controller or service logic fails midway through a workflow.

#### 6.5.5 Data Integrity Controls

The implemented database schema applies integrity controls at several levels: database constraints, ORM mappings, repository methods, and transaction handling. Database-level constraints are especially important because they protect the data even if a service or controller contains a defect.

| Integrity Control | Implemented Location | Purpose |
| --- | --- | --- |
| Unique `Username` | `User.Username` | Prevents duplicate usernames during registration or staff user creation. |
| Unique `Email` | `User.Email` | Prevents duplicate email identities. |
| Role enum | `User.Role` | Restricts persisted roles to `User`, `Moderator`, and `Admin`. |
| Status enum | `User.Status` | Restricts account status to `Enabled`, `Disabled`, and `Pending`. |
| Foreign key `Blueprint.UserID` | `Blueprint` table | Ensures each Blueprint belongs to an existing user. |
| Unique `(UserID, Name, Version)` | `Blueprint` table | Prevents duplicate Blueprint version labels for the same owner and name. |
| Check `ParentID IS NULL OR ParentID <> BlueprintID` | `Blueprint` table | Prevents a Blueprint from being its own parent in version lineage. |
| Foreign keys `Experiment.UserID` and `Experiment.BlueprintID` | `Experiment` table | Ensures each experiment belongs to a user and references an existing Blueprint. |
| Check `TrainSplit + ValSplit + TestSplit = 1.00` | `Experiment` table | Enforces complete split allocation. |
| Check `ValSplit >= 0.10 AND TestSplit >= 0.10` | `Experiment` table | Enforces minimum validation and test split sizes. |
| Foreign key `Model.ExperimentID` | `Model` table | Ensures each model belongs to an existing experiment. |
| Foreign keys `ExperimentLog.ExperimentID` and `ExperimentLog.ModelID` | `ExperimentLog` table | Ensures log rows are linked to valid experiment and model records. |
| Composite primary key `(UserID, ModelID)` | `FavoriteModel` table | Prevents duplicate model favorites by the same user. |
| Composite primary key `(UserID, BlueprintID)` | `FavoriteBlueprint` table | Prevents duplicate Blueprint favorites by the same user. |
| Primary key `BTCUSDTKline.Timestamp` | `BTCUSDTKline` table | Prevents duplicate market candles for the same timestamp. |

Market-data duplicate prevention is handled through an upsert in `MarketDataRepository.add()`. If a `BTCUSDTKline` record with the same `Timestamp` already exists, the OHLCV fields are updated rather than inserting a duplicate row.

```python
def add(kline):
    values = {
        "Timestamp": kline.Timestamp,
        "Open": kline.Open,
        "High": kline.High,
        "Low": kline.Low,
        "Close": kline.Close,
        "Volume": kline.Volume,
        "CreatedAt": kline.CreatedAt,
    }

    insert BTCUSDTKline(values)
    on conflict Timestamp:
        update Open, High, Low, Close, Volume, CreatedAt
```

These tests ensure that the implemented database remains aligned with RFC-002 and the Chapter 4 ERD as the rest of the BEE system continues to evolve.
